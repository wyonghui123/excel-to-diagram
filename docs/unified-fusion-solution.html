零、融合核心目标与预期价值

0.1 核心目标

消除BO图与SM图在代码层面的不对称性，建立统一的图表构建管线，使两种图表类型成为同一管线的两种输出模式，而非两套独立实现。

具体目标：

  • 统一标识策略：所有节点使用code作为唯一标识（id=code），消除name/code双标识的复杂度
  • 统一颜色计算：颜色在构建层统一计算，写入node.color，渲染层仅做格式转换
  • 统一构建接口：使用配置对象模式替代15+个独立参数，两种图共享同一构建函数签名
  • 统一分组模型：SM图直接使用ChartType.SERVICE_MODULE构建3层分组，消除BO类型workaround
  • 统一Store交互：两种图都通过相同机制回写layoutControlConfig，状态管理策略一致

0.2 预期价值

维度当前状态融合后价值量化
代码重复2套独立构建器（116+464行）1套统一构建器（~350行）减少约230行重复代码
颜色一致性双路径独立计算，可能不一致单路径计算，保证一致消除1类一致性bug
参数遗漏风险15个独立参数，已出过遗漏bug1个配置对象，类型安全消除1类参数遗漏bug
新增图表类型成本需新建独立Builder + Syntax（~500行）仅需新增ChartTypeConfig + 适配器（~100行）开发成本降低80%
调试定位时间需在2套代码中追踪数据流1套管线，追踪路径清晰调试时间减少50%

二、服务模块间的接口规范与通信协议

2.1 统一构建器接口（DiagramBuilder）

统一图表构建器 — 两种图共享同一构建入口
export class DiagramBuilder {
  constructor(private config: DiagramBuildConfig) {}
  
  build(): UnifiedDiagramData {
    const { chartType, data, scope, style, layout } = this.config
    
    // 1. 统一过滤（共享逻辑）
    const filteredData = this.filterByScope(data, scope)
    
    // 2. 统一颜色计算（共享逻辑）
    const colorContext = this.computeColors(filteredData, style, scope)
    
    // 3. 按图表类型构建节点（分化点）
    const nodes = this.buildNodes(filteredData, colorContext, scope)
    
    // 4. 按图表类型构建连线（分化点）
    const links = this.buildLinks(filteredData, scope)
    
    // 5. 统一容器构建（共享逻辑）
    const containers = this.buildContainers(filteredData)
    
    return {
      chartType,
      nodes,
      links,
      containers,
      ...style,
      ...layout
    }
  }
  
  // 共享方法
  private filterByScope(data, scope): FilteredData
  private computeColors(data, style, scope): ColorContext
  private buildContainers(data): UnifiedContainer[]
  
  // 分化方法（由chartType决定行为）
  private buildNodes(data, colorContext, scope): UnifiedDiagramNode[]
  private buildLinks(data, scope): UnifiedDiagramLink[]
}

2.2 颜色计算统一接口

ColorCalculator — 统一颜色计算，消除双路径问题
export class ColorCalculator {
  // 统一入口：构建层计算颜色
  static compute(config: {
    nodes: Array<{ code: string; domain: string; subDomain: string; isCenter: boolean }>
    colorGroupBy: string
    colorScheme: string
    centerScopeColor: string
    customColors: object
  }): Map<string, string> {
    const { nodes, colorGroupBy, colorScheme, centerScopeColor, customColors } = config
    const colors = COLOR_SCHEMES[colorScheme] || COLOR_SCHEMES.default
    const colorMap = new Map<string, string>()
    
    // 1. 按colorGroupBy分组
    const groupKeys = new Map<string, string>()
    nodes.forEach(node => {
      const key = colorGroupBy === 'subDomain' ? node.subDomain 
                : colorGroupBy === 'serviceModule' ? node.serviceModule 
                : node.domain
      groupKeys.set(node.code, key)
    })
    
    // 2. 为每个分组分配颜色
    const uniqueGroups = [...new Set(groupKeys.values())]
    const groupColorMap = new Map<string, string>()
    uniqueGroups.forEach((group, index) => {
      groupColorMap.set(group, customColors[group] || colors[index % colors.length])
    })
    
    // 3. 逐节点计算最终颜色
    nodes.forEach(node => {
      const baseColor = groupColorMap.get(groupKeys.get(node.code)) || colors[0]
      const finalColor = node.isCenter ? centerScopeColor : baseColor
      colorMap.set(node.code, finalColor)
    })
    
    return colorMap
  }
}

四、各模块间的依赖关系与集成策略

4.1 当前依赖关系 vs 融合后依赖关系

当前依赖（不对称）

  • BO图：diagramDataBuilder(116行) → useBusinessObjectSyntax(1073行)
  • SM图：serviceModuleDiagramBuilder(464行) → useServiceModuleSyntax(437行)

  • Store：统一回写layoutControlConfig
  • 标识：统一id=code

5.4 核心结论

融合的本质是"从两套管线回归一套管线"。

当前系统中BO图和SM图虽然共享了部分基础设施（GroupModel、diagramConfigStore、architectureProcessor），但在最关键的构建层和渲染层存在严重的职责分配不对称：BO图是"瘦构建+胖渲染"，SM图是"胖构建+瘦渲染"。这种不对称导致了颜色双路径计算、标识策略不一致、参数遗漏等一系列问题。

融合方案的核心是建立统一图表构建管线，将两种图的共同逻辑（过滤、颜色计算、分组模型、容器构建）提升到共享层，仅将节点/连线构建和渲染语法生成作为分化点。这样既保留了两者的功能差异，又消除了代码重复和一致性风险。

5个统一化要点：

  • 统一标识：id=code，消除name/code双标识
  • 统一颜色：ColorCalculator构建层计算，渲染层直接使用
  • 统一接口：DiagramBuildConfig配置对象，替代15+参数
  • 统一分组：SM图直接用SERVICE_MODULE类型，消除workaround
  • 统一状态：两种图统一回写Store，状态管理一致

渐进式实施策略：4个Phase按优先级递减实施，每个Phase独立可验证，不依赖后续Phase。Phase 1（统一标识）风险最高但价值最大，应优先实施；Phase 5（清理收尾）风险最低，最后实施。整体预计减少约230行重复代码，消除2类一致性bug，新增图表类型开发成本降低80%。
