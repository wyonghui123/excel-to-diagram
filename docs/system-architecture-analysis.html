一、数据模型：核心实体、属性定义、关系结构与约束条件

1.1 实体全景分类体系

系统数据实体按生命周期和功能定位分为五大类：原始实体（Excel解析直接产出，应用级生命周期）、推导实体（算法计算产出，请求级生命周期）、计算实体（响应式计算产出，响应式生命周期）、配置实体（用户交互产出，会话级生命周期）、输出实体（最终渲染消费，请求级生命周期）。两种图表类型共享原始实体，在推导、计算、配置和输出阶段产生分化。

分类实体生命周期图表类型存储位置持久性
原始实体（Excel解析产出 — 一对一映射，无数据损失）
原始BusinessObject应用级（随Excel加载创建）BO图 + SM图previewData.businessObjects随应用销毁
原始ServiceModule应用级SM图（主）/ BO图（辅）previewData.serviceModules随应用销毁
原始DomainProduct应用级BO图 + SM图previewData.domainProducts随应用销毁
原始Relationship应用级BO图（主）/ SM图（推导源）previewData.relationships随应用销毁
推导实体（算法计算产出 — N:1聚合，存在信息损失）
推导ServiceModuleRelationship请求级SM图局部变量不持久化
推导centerServiceModuleCodes请求级SM图局部变量不持久化
推导finalBoCodes请求级BO图 + SM图局部变量不持久化
计算实体（响应式计算产出 — 自动更新，依赖追踪）
计算CenterScopeMarkers响应式（watch centerScope）BO图 + SM图diagramConfigStorePinia持久
计算filteredContainers响应式（computed）BO图 + SM图useDiagramData不持久化
计算relationFilteredBoCodes响应式（computed）BO图 + SM图useDiagramData不持久化
配置实体（用户交互产出 — 会话级，可序列化）
配置DiagramConfig会话级BO图 + SM图diagramConfigStorePinia持久
配置LayoutControlConfig会话级BO图 + SM图diagramConfigStorePinia持久
输出实体（最终渲染消费 — 请求级，不可变）
输出Group（统一分组）请求级BO图 + SM图GroupModel.groups Map不持久化
输出DiagramData请求级BO图 + SM图useDiagramData.diagramData不持久化
输出DiagramNode请求级BO图 + SM图diagramData.nodes[]不持久化

1.2 核心实体属性定义与约束

BusinessObject（业务对象）原始实体
{
  code: string              PK NOT NULL UNIQUE
  name: string                     NOT NULL
  serviceModule: string            FK→SM.code FK
  serviceModuleName: string        冗余 可由 serviceModule→SM.name 推导
  subDomain: string                冗余 可由 serviceModule→SM.subDomain 推导
  domain: string                   冗余 可由 serviceModule→SM.domain 推导
  isCenter: boolean                计算 generateDiagram时赋值，默认false
  annotationCategory: string       ENUM: info|warning|error
  annotationContent: string
}
来源：Excel Sheet「业务对象」→ parseBusinessObjects() | 存储：previewData.businessObjects
约束：code全局唯一标识业务对象；serviceModule必须引用已存在的SM.code（FK完整性）；subDomain/domain为冗余派生字段，由Excel直接提供而非运行时推导，确保数据一致性需依赖Excel数据质量；isCenter仅在generateDiagram时计算赋值，不在原始数据中持久化
ServiceModule（服务模块）原始实体
{
  code: string              PK NOT NULL UNIQUE
  name: string                     NOT NULL
  subDomain: string                FK→SubDomain.name FK
  domain: string                   冗余 可由 subDomain→Domain 推导
  isCenter: boolean                计算 generateDiagram时赋值
  annotationCategory: string       ENUM: info|warning|error
  annotationContent: string
}
来源：Excel Sheet「服务模块」→ parseServiceModules() | 存储：previewData.serviceModules
约束：code全局唯一；subDomain必须引用DomainProduct.modules[].name；domain冗余，可通过subDomain→Domain推导；isCenter在generateDiagram中基于"SM是否包含centerScope中的BO"计算
DomainProduct（领域产品树）原始实体
{
  name: string              PK NOT NULL
  code: string                     UNIQUE
  isCenter: boolean                计算 generateDiagram时赋值
  modules: SubDomain[]             1:N
}

SubDomain {
  name: string              PK(局部)
  code: string
  isCenter: boolean                计算
  submodules: ServiceModuleRef[]   1:N
}

ServiceModuleRef {
  name: string
  code: string                     FK→SM.code
  isCenter: boolean                计算
  businessObjects: BORef[]         1:N
}
来源：Excel Sheet「领域产品」→ buildDomainProducts() | 存储：previewData.domainProducts
约束：三层嵌套结构 Domain→SubDomain→SM→BO；modules[].name领域内唯一；submodules[].code全局唯一；isCenter在generateDiagram中自底向上计算（BO→SM→SubDomain→Domain，全部isCenter时父级才isCenter）
Relationship（业务对象关系）原始实体
{
  sourceCode: string        FK→BO.code NOT NULL
  sourceName: string               冗余，可由 sourceCode→BO.name 推导
  targetCode: string        FK→BO.code NOT NULL
  targetName: string               冗余
  relationCode: string      PK UNIQUE
  relationDesc: string
  annotationCategory: string       ENUM: info|warning|error
  annotationContent: string
}
来源：Excel Sheet「业务对象关系」→ parseRelationships() | 存储：previewData.relationships
约束：sourceCode≠targetCode（无自环）；relationCode全局唯一；sourceCode和targetCode必须引用已存在的BO.code
ServiceModuleRelationship（服务模块关系）推导实体
{
  sourceServiceModuleCode: string   FK→SM.code
  sourceServiceModuleName: string   冗余
  targetServiceModuleCode: string   FK→SM.code
  targetServiceModuleName: string   冗余
  serviceRelationshipCode: string   PK UNIQUE
  businessObjectRelationshipCodes: string[]  聚合来源
  annotationContent: string         聚合标注
  annotationCategory: string        ENUM
}
来源：computedServiceModuleRelations() 推导 | 存储：局部变量
约束：source≠target（同SM内BO关系被丢弃）；PK格式=sourceCode-targetCode；从BO关系N:1聚合推导，非原始数据；信息损失：同SM内关系被丢弃、BO关系细节被聚合为标注
CenterScopeMarkers（中心范围标记）计算实体
{
  domains: Map<string, boolean>
    // key: 领域名称
    // value: 该领域是否包含中心BO（true/false）
  subDomains: Map<string, boolean>
    // key: 子领域名称
    // value: 该子领域是否包含中心BO
  serviceModules: Map<string, true>
    // key: SM.name 或 SM.code（同时存储两种形式）
    // value: 固定为 true
}
来源：updateCenterScopeMarkers() | 存储：diagramConfigStore.centerScopeMarkers
约束：仅基于centerScope计算，不含relationFilteredBoCodes；serviceModules Map的key同时包含name和code两种形式（解决不同组件使用不同标识符的问题）；domains/subDomains的value为boolean（表示是否包含），serviceModules的value固定为true（仅表示存在性）
Group（统一分组）输出实体
{
  id: string                PK  格式: {Type}_{code}
  type: GroupType           ENUM: DOMAIN|SUB_DOMAIN|
                            SERVICE_MODULE|BUSINESS_OBJECT|LAYOUT
  title: string             显示标题
  elementRef: ElementRef | null  架构元素引用
  parentId: string | null   FK→Group.id
  children: Group[]         1:N 子分组
  layout: {
    direction: string       ENUM: LR|TB|RL|BT
    visible: boolean        边框可见性
    enabled: boolean        分组启用状态
    style: { fill, stroke, strokeWidth, strokeDasharray }
  }
  containers: ContainerNode[]  配置阶段节点列表
}

ElementRef {
  type: GroupType           引用的架构元素类型
  code: string              架构元素编码
  name: string              架构元素名称
  parentCode: string | null 父级编码
  grandparentCode: string | null  祖父级编码
}
来源：buildGroupModelFromArchitecture() | 存储：GroupModel.groups Map
约束：id格式由createGroupId()生成（如D_供应链、SD_采购管理、SM_CGGL）；elementRef可选，自定义分组无elementRef；BO图层级：DOMAIN→SUB_DOMAIN→SERVICE_MODULE→BUSINESS_OBJECT（4层）；SM图层级：DOMAIN→SUB_DOMAIN→SERVICE_MODULE（3层，SERVICE_MODULE为末端节点）
DiagramConfig（图表配置）配置实体
{
  chartType: string              ENUM: businessObject|serviceModule
  colorScheme: string            ENUM: default|vibrant|pastel|warm|cool|business|nature
  colorGroupBy: string           ENUM: domain|subDomain|serviceModule
  centerScopeColor: string       HEX颜色，默认 #EDEDED
  centerDomainColor: string      HEX颜色，默认 #D9D9D9
  nodeTextColor: string          HEX颜色，默认 black
  centerScope: string[]          中心范围BO编码数组
  centerScopeMarkers: CenterScopeMarkers
  layoutTemplate: string         ENUM: default|grid|horizontal|vertical
  layoutEngine: string           ENUM: elk|dagre
  layoutControlConfig: object    分组控制配置
  customColors: object           自定义颜色映射
  hideLinkLabelTails: boolean | null
  assignmentMode: string         ENUM: auto|manual
}
来源：用户交互 | 存储：diagramConfigStore (Pinia)
约束：chartType决定数据流分支（BO图 vs SM图）；colorGroupBy决定颜色计算策略；centerScope为空时centerScopeMarkers也应为空；layoutControlConfig由GroupModel.toMermaidConfig()生成后回写Store
DiagramNode（图表节点）输出实体
// BO图节点（diagramDataBuilder.js → buildNodes）
{
  id: string            // bo.name（注意：用name而非code）
  name: string
  code: string          // bo.code PK
  domain, subDomain, serviceModule, serviceModuleName
  category: 'object'
  isCenter: boolean     // generateDiagram时赋值
  // 注意：color 未在构建时计算，由渲染层动态计算
}

// SM图节点（serviceModuleDiagramBuilder.js）
{
  id: string            // sm.code
  name: string
  code: string          // sm.code PK
  domain, subDomain
  color: string         构建时计算
  textColor: string     构建时计算
  isCenter: boolean     构建时计算
  annotationCategory, annotationContent
}
来源：buildDiagramData() / buildServiceModuleDiagramData()
关键差异：①BO图节点id=bo.name，SM图节点id=sm.code（标识策略不同）；②SM图节点在构建时已计算color和isCenter，BO图节点未计算color（颜色在渲染时由Mermaid组件动态计算）；③BO图节点有serviceModule/serviceModuleName字段，SM图节点无
ChartTypeConfig（图表类型配置）配置实体
// businessObject 图表类型
{

// serviceModule 图表类型

1.5 约束条件汇总

约束类型约束名称适用实体规则描述违反后果
NOT NULL编码非空BO.code, SM.code, Domain.name主键字段不允许为空数据解析失败或节点无法标识
UNIQUE编码唯一BO.code, SM.code, Rel.relationCode主键字段全局唯一节点重复或关系冲突
FK外键引用BO.serviceModule→SM.code外键必须引用已存在的目标BO无法归属到SM，分组错误
ENUM枚举约束GroupType, colorGroupBy, chartType值必须在预定义集合内逻辑分支未覆盖，行为未定义
业务约束同SM内关系丢弃ServiceModuleRelationshipsource≠target同SM内BO关系不产生SM关系
业务约束isCenter自底向上Domain, SubDomain, SM子级全部isCenter时父级才isCenter领域/子领域颜色标记错误
语义约束nodes语义分化ContainerNodeBO图nodes=BO[]，SM图nodes=SM[]类型混淆导致运行时错误

2.2 各阶段数据转换规则

Stage 1: Excel解析 → previewData

转换规则：Excel Sheet → 结构化对象（一对一映射，无数据损失）
处理链：parseExcelFile() → parseServiceModules() → parseBusinessObjects() → parseRelationships() → buildServiceModules() → buildDomainProducts() → buildPreviewData()

Sheet「业务对象」  → BusinessObject[]   { code, name, serviceModule, serviceModuleName, subDomain, domain }
Sheet「服务模块」  → ServiceModule[]   { code, name, subDomain, domain }
Sheet「领域产品」  → DomainProduct[]   { name, code, modules: [{ name, code, submodules: [{ name, code, businessObjects }] }] }
Sheet「业务对象关系」→ Relationship[]   { sourceCode, targetCode, relationCode, relationDesc }

转换特点：
  - 一对一映射，无数据损失
  - 冗余字段（subDomain, domain）在BO中由Excel直接提供
  - DomainProduct构建三层嵌套：Domain→SubDomain→SM→BO
  - serviceModuleMap 和 moduleHierarchy 为中间产物，最终转为 serviceModules[] 和 domainProducts[]

Stage 2: 中心范围选择 → centerScope + centerScopeMarkers

转换规则：用户选择 → 标记映射（N:1聚合，存在信息损失）
输入：用户在 StepScope.vue 中勾选业务对象
输出：centerScope = ['BCDM01', 'BCDM02', ...]  // BO编码数组

推导：updateCenterScopeMarkers()
  centerScope → 遍历 businessObjects → 找到包含中心BO的SM
  → 遍历 domainProducts → 标记包含中心BO的领域/子领域
  输出：centerScopeMarkers = {
    domains: Map<领域名, boolean>,        // 是否包含中心BO
    subDomains: Map<子领域名, boolean>,   // 是否包含中心BO
    serviceModules: Map<SM.name|SM.code, true>  // 同时存储name和code
  }

转换特点：
  - N:1 聚合（多个BO编码 → 少量SM标记）
  - 信息损失（只保留是否包含，不保留具体哪些BO）
  - 仅基于 centerScope，不含 relationFilteredBoCodes
  - serviceModules Map 同时存储 name 和 code 两种 key（解决不同组件使用不同标识符的问题）
  - 使用 callId 防止过时更新（并发控制）

Stage 3: 关系范围 + 中心范围 → finalBoCodes

转换规则：并集计算（单调递增）
输入：centerScope (Set) + relationFilteredBoCodes (Set)
输出：finalBoCodes = centerScope ∪ relationFilteredBoCodes

关系分类路径：
  relationships → buildRelationCategoryTree() → relationCategoryTree (computed)
  → 用户选择关系节点 → getSelectedRelationCodes() → filteredRelations
  → 从 filteredRelations 提取 sourceCode/targetCode → relationFilteredBoCodes

关系分类维度（relationClassifier.js）：
  范围类型：internal（中心范围内）/ external（中心范围外）
  分类类型：cross-domain / same-domain-cross-subdomain / same-subdomain-cross-module / same-module

转换特点：单调递增（只增不减）；关系范围依赖用户选择的关系类别

Stage 4: SM关系推导

转换规则：BO关系 → SM关系（N:1聚合，存在信息损失）
输入：Relationship[] + BusinessObject[] + ServiceModule[]
处理（computedServiceModuleRelations）：
  1. 构建 boToModuleMap: bo.code → { moduleCode, moduleName }
  2. 构建 moduleCodeToNameMap: sm.code → sm.name
  3. 遍历 Relationship，按 (sourceModule, targetModule) 分组
  4. 跳过 sourceModule === targetModule（同SM内关系丢弃）
  5. 聚合 BO 关系编码和标注
输出：ServiceModuleRelationship[]

转换特点：
  - N:1 聚合（多条BO关系 → 一条SM关系）
  - 信息损失：同SM内关系被丢弃、BO关系细节被聚合为标注
  - 标注去重：多个BO关系标注用分号连接

Stage 5: filteredContainers 构建（图表类型分化点）

BO图路径
DomainProduct[] → 按 finalBoCodes 过滤
→ 按 subDomain 分组为容器

container = {
  id: module.code || module.name,
  name: module.name,
  fullTitle: domain.name / module.name,
  domain, domainCode, subDomainName,
  nodes: BusinessObject[],  // ← BO节点
  serviceModuleMap: { ... }  // SM→BO映射
}

过滤条件：BO.code ∈ finalBoCodes
SM图路径
ServiceModule[] → 按 finalBoCodes 过滤
（检查SM的BO是否在范围内）
→ 按 subDomain 分组为容器

container = {
  id: sm.subDomain,
  name: sm.subDomain,
  fullTitle: domain.name / sm.subDomain,
  domain, domainCode, subDomainName,
  nodes: ServiceModule[]  // ← SM节点
}

过滤条件：SM的任一BO.code ∈ finalBoCodes
[WARNING] 语义歧义（系统最关键的设计歧义点）：container.nodes 在 BO 图中存储 BusinessObject[]，在 SM 图中存储 ServiceModule[]。同一字段名、不同数据类型和语义。此歧义在 handleServiceModuleAutoGroup() 中导致过"采购供应分组下是采购供应节点"的bug——将容器本身误当作节点添加到分组中。

Stage 6: 统一分组模型构建

转换规则：架构数据 → Group树（由 ChartTypeConfig 驱动分化）
处理链：buildGroupModelFromArchitecture(architectureData, chartType)

BO图（buildBusinessObjectGroupModel）：
  DomainProduct[] → 遍历 domain → createGroup(DOMAIN)
    → 遍历 subDomain → createGroup(SUB_DOMAIN)
      → 遍历 sm → createGroup(SERVICE_MODULE)
        → 遍历 bo → createGroup(BUSINESS_OBJECT)  // 末端节点
  层级：DOMAIN → SUB_DOMAIN → SERVICE_MODULE → BUSINESS_OBJECT（4层）

SM图（buildServiceModuleGroupModel）：
  DomainProduct[] → 遍历 domain → createGroup(DOMAIN)
    → 遍历 subDomain → createGroup(SUB_DOMAIN)
      → 遍历 sm → createGroup(SERVICE_MODULE)  // 末端节点
  层级：DOMAIN → SUB_DOMAIN → SERVICE_MODULE（3层）

每个 Group 通过 elementRef 关联架构元素：
  elementRef = { type, code, name, parentCode?, grandparentCode? }

GroupModel.fromUserConfig(architectureGroups, userConfig, chartType)：
  1. 用 architectureGroups 构建 GroupModel 索引（Map<id, Group>）
  2. 递归合并 userConfig.groups（按 id / elementCode / title 匹配）
  3. 合并内容：layout（direction/visible/enabled/style）、containers

GroupModel.toMermaidConfig()：
  1. getFlattenedGroups()：处理禁用分组（提升子分组到根级别）
  2. convertGroup()：递归转换为旧版配置格式
  3. 输出：{ enabled, groups, titleMap }

转换特点：
  - 统一模型：BO图和SM图使用相同的Group结构和GroupModel类
  - ChartTypeConfig驱动：terminalTypes决定末端节点类型
  - 用户配置合并：支持按id/elementCode/title三种方式匹配
  - 禁用提升：禁用的分组其子分组被提升到父级

Stage 7: 图表构建（图表类型分化点）

BO图：buildDiagramData()
输入：filteredBusinessObjects, finalRelationships,
      filteredDomainProducts, filteredServiceModules,
      colorGroupBy, centerScopeColor, colorScheme,
      nodeTextColor, centerScope, layoutControlConfig, ...

处理：
  buildNodes(businessObjects) → nodes[]
    - id = bo.name（注意：用name而非code）
    - isCenter = centerScopeSet.has(bo.code)
    - color 未计算（渲染时动态计算）
  buildLinks(relationships) → links[]

输出：{ nodes, links, domainProducts, serviceModules,
        colorGroupBy, centerScopeColor, centerScope,
        layoutControlConfig, ... }
SM图：buildServiceModuleDiagramData()
输入：filteredServiceModules, filteredSMRelationships,
      filteredDomainProducts, centerScopeColor,
      colorGroupBy, colorScheme, centerServiceModuleCodes,
      layoutControlConfig, ...

处理：
  1. finalCenterServiceModuleCodes = centerServiceModuleCodes
     ? new Set(centerServiceModuleCodes)
     : 从 sm.isCenter 或 centerSubDomain 推导
  2. 颜色映射：
     subDomain模式：subDomainColors[subDomain] = color
     domain模式：domainColors[domain] = color
  3. 节点构建：
     isCenter = finalCenterServiceModuleCodes.has(sm.code)
     baseColor = subDomainColors[subDomain] || domainColors[domain]
     finalColor = isCenter ? centerScopeColor : baseColor

输出：{ nodes[], links[], containers[],
        centerSubDomain, colorGroupBy, layoutControlConfig, ... }

2.3 颜色计算双路径分析

颜色计算是系统中唯一存在双路径独立计算的关键逻辑，两条路径可能产生不一致的结果。

路径A：配置页面（GroupItem.vue）
触发：配置页面渲染每个容器/节点时
函数：getContainerColor(container)

判断逻辑：
  1. 检查 container.code/elementCode ∈ centerScope
  2. 检查 container.name ∈ centerScope
  3. 检查 container.name/code ∈
     centerScopeMarkers.serviceModules
  → 若命中任一：返回 centerScopeColor
  → 否则：按 colorGroupBy 计算 baseColor

特点：
  - 基于 centerScopeMarkers（响应式更新）
  - 逐节点判断（精确到单个节点）
  - 不依赖 domain/subDomainName 级别判断
路径B：图表构建（serviceModuleDiagramBuilder.js）
触发：generateDiagram() 调用时
函数：buildServiceModuleDiagramData()

判断逻辑：
  1. finalCenterServiceModuleCodes =
     centerServiceModuleCodes ? new Set(...)
     : 从 sm.isCenter 或 centerSubDomain 推导
  2. isCenter = finalCenterServiceModuleCodes.has(sm.code)
  3. baseColor = colorGroupBy === 'subDomain'
     ? subDomainColors[sm.subDomain]
     : domainColors[sm.domain]
  4. finalColor = isCenter ? centerScopeColor : baseColor

特点：
  - 基于 centerServiceModuleCodes（请求时计算）
  - 逐节点判断（精确到单个SM）
  - 依赖 colorGroupBy 决定 baseColor 来源
[WARNING] 双路径一致性风险：路径A使用 centerScopeMarkers.serviceModules（基于centerScope通过BO→SM桥接推导），路径B使用 centerServiceModuleCodes（同样基于centerScope通过BO→SM桥接推导）。两者数据源相同但计算时机不同——路径A是响应式的，路径B是请求时的。如果 centerScope 在两次计算之间发生变化，可能导致配置页面和图表展示的颜色不一致。历史上 centerScopeColor 未传入 buildServiceModuleDiagramData() 就是路径B遗漏参数的典型案例。

2.5 数据流关键路径追踪

2.5.1 centerScopeColor 传递链（历史bug案例）

centerScopeColor 从用户选择到节点颜色的完整传递链
用户选择颜色 → diagramConfigStore.centerScopeColor (ref)
  → useDiagramData.diagramConfig (computed)
    → generateDiagram() 读取 diagramConfig.value.centerScopeColor
      → buildServiceModuleDiagramData({ centerScopeColor: diagramConfig.value.centerScopeColor })
        → finalColor = isCenter ? centerScopeColor : baseColor
          → diagramData.nodes[].color

同时（配置页面路径）：
  diagramConfigStore.centerScopeColor
    → GroupItem.vue centerScopeColor (computed from store)
      → getContainerColor() 返回 centerScopeColor

历史bug：centerScopeColor 未传入 buildServiceModuleDiagramData()，
导致SM图中心节点使用默认颜色而非用户选择的中心范围颜色。
根因：函数参数列表过长（15+个参数），新增参数时遗漏传递。

2.5.2 centerServiceModuleCodes 推导链

从 centerScope（BO编码）到 centerServiceModuleCodes（SM编码）的推导过程
centerScope = ['BCDM01', 'BCDM02', ...]  // BO编码数组
  → centerBoCodes = new Set(centerScope)
    → filteredBusinessObjects.filter(bo => centerBoCodes.has(bo.code))
      → 遍历 filteredBO，收集 bo.serviceModule
        → centerServiceModuleCodes = new Set([bo.serviceModule...])

推导特点：
  - N:1 聚合（多个BO编码 → 少量SM编码）
  - 依赖 bo.serviceModule 字段（FK桥接）
  - 仅在 generateDiagram() 中计算，非响应式
  - 与 centerScopeMarkers.serviceModules 使用相同的数据源和推导逻辑
    但计算时机和存储方式不同

三、数据模型与数据流的关键差异分析

3.1 本质特征对比：静态结构 vs 动态过程

[SYMBOL]数据模型 — 静态结构

  • 本质：描述数据的"形状"——有哪些实体、属性、关系
  • 关注点：数据是什么、如何组织、约束条件
  • 时间维度：静态快照，不随时间变化（除非Schema变更）
  • 验证方式：类型检查、约束校验、关系完整性
  • 变更频率：低（Schema稳定，仅新增字段）
  • 文档形式：ER图、数据字典、TypeScript类型定义
  • 错误表现：编译时/解析时即可发现（类型不匹配、字段缺失）

[SYMBOL]数据流 — 动态过程

  • 本质：描述数据的"运动"——如何产生、传递、转换、消费
  • 关注点：数据从哪来、到哪去、经过什么处理
  • 时间维度：动态过程，随用户操作实时变化
  • 验证方式：端到端一致性、转换正确性、时序正确性
  • 变更频率：高（业务流程调整、组件重构）
  • 文档形式：流程图、序列图、数据流图
  • 错误表现：运行时才暴露（参数遗漏、时序错误、数据不一致）

3.2 功能定位差异

对比维度数据模型数据流
核心问题 数据"长什么样"？——实体结构、属性类型、关系约束 数据"怎么跑"？——流转路径、处理步骤、转换规则
功能定位 定义数据的存储结构和组织方式 定义数据的处理逻辑和传递方式
表现形式 TypeScript interface / JSON Schema / 类型定义 函数调用链（computed → watch → function call chain）
存储特性 数据在特定位置"停留"：previewData、diagramConfigStore、GroupModel.groups 数据在组件间"穿越"：Props、Emit、Store、函数参数
一致性保障 类型系统（TypeScript）、约束条件（PK/FK/NOT NULL/UNIQUE） 响应式系统（Vue computed/watch）、调用顺序、参数传递完整性
错误类型 结构错误：字段缺失、类型不匹配、约束违反、FK引用断裂 流程错误：参数遗漏、调用时序错误、数据不一致、响应式依赖缺失
设计考量 规范化（减少冗余）vs 性能（适当冗余如 sm.domain、bo.subDomain） 单向数据流（可预测性）vs 双向通信（灵活性）
演化规律 增量式：新增字段/实体，向后兼容，旧字段可标记deprecated 重构式：流程调整可能改变整条数据链路，牵一发动全身
测试策略 单元测试：验证数据结构、约束、类型、FK完整性 集成测试：验证端到端数据一致性、转换正确性、参数传递完整性
维护关注点 Schema迁移、向后兼容、文档同步、冗余字段一致性 调用链完整性、参数传递正确性、响应式依赖正确性、时序一致性

3.3 数据存储 vs 数据传输的特性差异

[SYMBOL]数据存储特性（数据模型视角）

  • 持久性：previewData随应用生命周期存在；diagramConfigStore通过Pinia持久化
  • 结构稳定性：BusinessObject、ServiceModule等实体结构在Excel解析后不再变化
  • 冗余策略：bo.subDomain、bo.domain、sm.domain为性能冗余，避免运行时多跳查询
  • 标识唯一性：code字段全局唯一，作为PK和FK的基础
  • 嵌套深度：DomainProduct三层嵌套，GroupModel最多四层嵌套
  • 类型安全：TypeScript类型定义提供编译时检查

[SYMBOL]数据传输特性（数据流视角）

  • 瞬时性：函数参数、局部变量在调用栈中存在，调用结束即销毁
  • 转换动态性：finalBoCodes、centerServiceModuleCodes每次generateDiagram重新计算
  • 信息损失：SM关系推导丢弃同SM内BO关系；centerScopeMarkers只保留boolean标记
  • 传递链脆弱性：centerScopeColor需经4级传递，任一环节遗漏即导致bug
  • 路径分叉：BO图和SM图在Stage 5后走不同处理路径
  • 一致性挑战：颜色双路径独立计算，可能产生不一致结果

3.5 演化规律对比

[DECORATIVE]数据模型演化规律

  • 增量式增长：新增字段（如isCenter、annotationCategory）不影响现有结构
  • 向后兼容：旧字段可标记deprecated但保留（如serviceModuleName）
  • 统一化趋势：从分散的ContainerNode/GroupModel向统一Group模型演化
  • 冗余收敛：新模型通过elementRef减少冗余，但旧模型冗余字段仍存在
  • 类型强化：GroupType枚举替代字符串groupType，增强类型安全
  • 演化风险低：结构变更影响范围可控，可通过类型系统检测

[REFRESH]数据流演化规律

  • 重构式变更：新增参数需修改整条调用链（如centerScopeColor遗漏）
  • 路径分叉加剧：BO图和SM图的处理差异随功能增加而扩大
  • 双路径累积：颜色计算等逻辑在不同组件中独立实现，难以统一
  • 隐式依赖增加：响应式watch/computed的依赖关系隐式，难以追踪
  • 参数膨胀：buildServiceModuleDiagramData参数从7个增长到15+个
  • 演化风险高：流程变更可能影响整条数据链路，难以预测影响范围

3.6 开发维护中的不同关注点

[SYMBOL]数据模型维护关注点

  • 新增字段时：确保类型定义更新、约束条件添加、文档同步
  • 修改关系时：检查FK引用完整性、级联影响范围
  • 重构时：向后兼容性检查、数据迁移方案
  • 典型风险：字段语义漂移（如nodes字段含义随图表类型变化）
  • 验证手段：TypeScript编译检查、单元测试、数据校验（dataValidator.js）
  • 修复成本：低——结构问题通常可通过类型系统快速定位

[SYMBOL]数据流维护关注点

  • 新增参数时：确保整条调用链都传递新参数（如centerScopeColor遗漏）
  • 修改流程时：验证端到端数据一致性、响应式依赖正确性
  • 重构时：调用顺序可能改变、中间状态可能不一致
  • 典型风险：参数传递链断裂（中间环节遗漏关键参数）
  • 验证手段：端到端集成测试、数据流日志（DataFlowLogger）、运行时断言
  • 修复成本：高——流程问题通常需要追踪整条数据链路才能定位

3.7 核心结论

数据模型是"地图"，数据流是"路线"。

数据模型定义了系统中存在哪些"地点"（实体）、它们有哪些"特征"（属性）、彼此如何"相邻"（关系）。数据流定义了数据如何在这些"地点"之间"旅行"——经过哪些"中转站"（组件）、采用什么"交通工具"（Props/Store/函数调用）、在中转站做什么"加工"（转换规则）。

本系统中最频繁的bug源于数据流问题，而非数据模型问题。数据模型相对稳定，一旦定义好很少变动；数据流则随业务逻辑调整而频繁变化，且调用链越长、分叉越多，出错概率越高。centerScopeColor未传入、颜色双路径计算不一致、变量声明顺序错误、centerScopeMarkers多次重建——这些都是数据流问题，不是数据模型问题。

静态结构与动态过程的本质差异：数据模型的静态性使其可以通过类型系统在编译时捕获错误；数据流的动态性使其错误只能在运行时暴露。数据模型的冗余字段（如bo.domain）是设计选择，有明确的权衡；数据流的参数遗漏（如centerScopeColor）是设计缺陷，没有合理的权衡。数据模型的演化是增量的、向后兼容的；数据流的演化是重构式的、可能破坏整条链路的。

优化建议：

  • 数据模型层面：统一container.nodes的语义（引入nodeType区分BO/SM）；统一节点标识策略（BO图id=bo.code而非bo.name）；消除centerScopeMarkers的双重标识
  • 数据流层面：缩短参数传递链（将配置对象整体传入而非逐个参数）；统一颜色计算路径（消除GroupItem.vue和serviceModuleDiagramBuilder.js的双路径）；增量更新centerScopeMarkers；为buildServiceModuleDiagramData引入配置对象模式替代15+个独立参数
  • 架构层面：引入数据流验证层（在关键转换节点添加断言，确保输入输出一致性）；建立参数传递完整性检查机制（函数签名变更时自动检测调用点）
