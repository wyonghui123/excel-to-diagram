1.2 实体间关系

关系类型说明连接字段
Domain → SubDomain 1:N 一个领域包含多个子领域 domain.modules[].name
SubDomain → ServiceModule 1:N 一个子领域包含多个服务模块 subDomain.submodules[].code
ServiceModule → BusinessObject 1:N 一个服务模块包含多个业务对象 bo.serviceModule = sm.code
ServiceModule ↔ ServiceModule N:N 服务模块间关系（从BO关系推导） SMRelationship.source/targetCode
BusinessObject ↔ BusinessObject N:N 业务对象间关系（原始数据） Relationship.source/targetCode
CenterScope → ServiceModule N:N 中心范围BO推导出的中心SM bo.serviceModule → sm.code

字段类型用途使用场景

sm.namestring服务模块显示名称节点标签、颜色key（serviceModule模式）

三、优势与不足分析

[OK] 优势

  • 数据源统一：业务对象和服务模块共享同一套 previewData 数据源，保证了数据一致性
  • 关系推导设计合理：SM 关系从 BO 关系自动推导，避免用户重复维护
  • 中心范围标记分层：centerScopeMarkers 分 domains/subDomains/serviceModules 三层，支持多种颜色分组模式
  • 过滤逻辑清晰：finalBoCodes = centerScope ∪ relationFiltered，范围只增不减
  • 分组模型灵活：groups 支持 domain → subDomain 两级嵌套，也支持扁平结构

[X] 不足

  • 颜色计算双路径：GroupItem.vue 和 serviceModuleDiagramBuilder.js 各自独立计算颜色，逻辑不一致风险高
  • 数据冗余：sm.domain 是冗余字段（可通过 subDomain 推导），BO 中也有大量冗余字段
  • centerScopeMarkers 多次重建：每次 centerScope 变化都完整重建 markers，无增量更新
  • SM关系推导丢失信息：同一SM内的BO关系被丢弃，无法反映模块内部结构
  • container.nodes 语义模糊：业务对象图中 nodes 是 BO，服务模块图中 nodes 是 SM，同一字段不同语义
  • centerScopeColor 传递链过长：从 diagramConfig → generateDiagram → buildServiceModuleDiagramData → 节点颜色，中间任一环节缺失都会导致颜色错误

3.1 关键设计问题

问题影响建议
颜色计算双路径 分组配置页面和图表展示颜色可能不一致 统一颜色计算逻辑，建议在 diagramData 构建时一次性计算
container.nodes 语义模糊 代码可读性差，容易引入 bug 区分 nodeType 或使用不同字段名（如 smNodes, boNodes）
centerScopeMarkers 重复计算 性能浪费，可能导致中间状态闪烁 使用增量更新或 debounce
SM关系推导不可逆 无法从SM关系反查具体BO关系 保留原始BO关系引用，支持展开查看
