1.2 代码量与复杂度对比

BO 业务对象图核心代码

  • diagramDataBuilder.js: 116行 — 极简，仅做字段映射
  • useBusinessObjectSyntax.js: 1073行 — 最复杂的渲染逻辑
  • buildNodes(): 14行 — 一对一字段映射
  • buildLinks(): 13行 — 一对一字段映射
  • buildDiagramData(): 34行 — 简单组装
  • 颜色计算: 渲染层（useBusinessObjectSyntax内）约80行
  • 分组模型: 4层层级（DOMAIN→SUB_DOMAIN→SM→BO）

SM 服务模块图核心代码

  • serviceModuleDiagramBuilder.js: 464行 — 包含颜色、过滤、排序
  • useServiceModuleSyntax.js: 437行 — 较简洁的渲染逻辑
  • buildServiceModuleDiagramData(): 180行 — 包含完整业务逻辑
  • 颜色计算: 构建层（builder内）约70行
  • 分组模型: 3层层级（DOMAIN→SUB_DOMAIN→SM）
  • 额外逻辑: extractServiceModuleCodesFromGroups() 约65行
  • 额外逻辑: sortContainersByTemplate() 约110行
[WARNING] 关键发现：BO图的数据构建层极简（116行），复杂度集中在渲染层（1073行）；SM图的数据构建层较复杂（464行），渲染层相对简洁（437行）。这反映了两种图在职责分配上的根本差异——BO图将颜色计算推迟到渲染时，SM图在构建时就完成了颜色计算。

三、业务逻辑实现方式差异

这是两种图在业务逻辑实现上最根本的差异。BO图将颜色计算推迟到Mermaid语法生成阶段（渲染层），SM图在数据构建阶段就完成了颜色计算。

4.3 architectureProcessor中的分支差异

buildBusinessObjectGroupModel()
层级：Domain → SubDomain → SM → BO（4层）

特点：
1. 使用 boMap（bo.code → bo对象）
2. SM层有 businessObjects 遍历
3. BO作为末端节点（GroupType.BUSINESS_OBJECT）
4. BO的elementRef包含parentCode（SM code）
5. 空SM（无BO）不会被添加到树中
6. 约100行代码

buildServiceModuleGroupModel()
层级：Domain → SubDomain → SM（3层）

7. 有大量console.log调试语句

4.4 Mermaid语法生成交互差异

useBusinessObjectSyntax.js
函数签名：
generateMermaidCode(
  data,                    // diagramData
  relationDescriptions,    // 输出参数
  layoutEngine,            // 'dagre' | 'elk'
  layoutType,              // 'grouped'

)  // 5个参数

数据消费方式：
- 从 data.nodes 读取节点（需要过滤 category === 'object'）
- 从 data.domainProducts 重新构建 objectToModuleMap
- 从 data.centerScope 读取中心范围
- 从 data.colorGroupBy 读取分组方式
- 颜色在函数内部计算（不使用 data.nodes[].color）

关键特点：
- 需要重新构建模块分组映射（约80行）
- 需要重新计算颜色（约40行）
- 支持连线颜色计算（约30行）
- 支持 virtualContainer 排序（约80行）
- 代码量：1073行
useServiceModuleSyntax.js
函数签名：
generateMermaidCode(
  data,                    // diagramData
  relationDescriptions,    // 输出参数

数据消费方式：
- 直接使用 data.nodes（无需过滤，全是SM节点）
- 直接使用 data.containers（已按subDomain分组）

- 直接使用 data.nodes[].isCenter（构建时已计算）

- 无需重新构建映射（直接使用containers）
- 无需计算颜色（构建时已完成）
- 支持 resolveGroupContainers（约120行）

- 代码量：437行（不到BO图的一半）

5.3 接口设计核心区别

5.4 核心结论

两种图的代码实现遵循了截然不同的职责分配哲学。

BO图 = "瘦构建 + 胖渲染"：diagramDataBuilder.js仅做116行的字段映射，将所有计算职责（颜色、分组、映射）推迟到useBusinessObjectSyntax.js（1073行）的渲染阶段。这种设计的优势是构建层简单可靠，劣势是渲染层过于复杂，且颜色计算结果无法被其他组件复用（如配置页面GroupItem.vue需要独立计算颜色）。

SM图 = "胖构建 + 瘦渲染"：serviceModuleDiagramBuilder.js承担了464行的计算职责（颜色、过滤、排序、中心判断），useServiceModuleSyntax.js仅做437行的格式转换。这种设计的优势是构建结果自包含（nodes[].color/isCenter已确定），渲染层简洁，且颜色结果可被其他组件复用；劣势是构建层逻辑复杂，参数列表过长（15个参数），容易遗漏（如centerScopeColor历史bug）。

最根本的5个差异点：

  • 标识策略：BO图用name作id（中文，需双映射），SM图用code作id（编码，单映射）——这是所有下游差异的根源
  • 颜色计算时机：BO图渲染层计算（不持久化），SM图构建层计算（写入node.color）——导致颜色双路径问题
  • 关系数据来源：BO图直接使用原始关系，SM图从BO关系推导——引入computedServiceModuleRelations()的额外复杂度
  • 分组模型构建：SM图使用ChartType.BUSINESS_OBJECT构建4层分组再用SERVICE_MODULE合并——这是一个workaround，暗示分组模型对SM图的支持不够原生
  • Store交互：SM图回写layoutControlConfig到Store，BO图不回写——导致两种图的状态管理策略不一致

统一化建议：

  • 统一标识策略：BO图也应使用code作id，消除双映射的复杂度
  • 统一颜色计算时机：两种图都应在构建层计算颜色，渲染层仅做格式转换
  • 统一构建函数签名：引入配置对象模式（如DiagramBuildConfig），替代15+个独立参数
  • 修复SM图分组构建workaround：使用ChartType.SERVICE_MODULE直接构建3层分组，而非先建4层再合并
  • 统一Store交互策略：两种图都应（或都不应）回写layoutControlConfig到Store
