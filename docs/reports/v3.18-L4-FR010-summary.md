## 一、L4 健康监控实现

### 1.1 service_manager.ps1 增强

**问题**: 之前 `service_manager.ps1 status` 只检查端口是否监听，不验证 PID 一致性，导致孤儿进程无法被发现。

**修复**:
- 添加 listener-PID 校验：通过 `Get-NetTCPConnection` 获取实际监听进程 PID
- 与 `.service_status_<port>.json` 中记录的 PID 对比
- 不一致时输出警告：`LISTENER MISMATCH: status.json PID=X but actual listener PID=Y (orphan process?)`

**验证**:

```powershell
  Backend (Waitress)        : RUNNING  (port=3010, pid=1556)   ⚠️ LISTENER MISMATCH: status.json PID=1556 but actual listener PID=15036
```

### 1.2 health_monitor.ps1 实现

**功能**: 后台监控脚本，每 5 秒 ping `/health` 端点，连续失败 2 次自动重启服务。

**核心逻辑**:
```powershell
# 每 5 秒检查一次
while ($true) {
    $healthy = Test-HealthEndpoint -Port $Port -Path $HealthPath -TimeoutSec $HealthTimeoutSec
    
    if ($healthy) {
        $failureCount = 0
    } else {
        $failureCount++
        if ($failureCount -ge $FailureThreshold) {
            Write-Host "[HEALTH] Service unhealthy, triggering restart..."
            & $PSScriptRoot\service_manager.ps1 restart
            $failureCount = 0
        }
    }
    
    Start-Sleep -Seconds $IntervalSec
}
```

**验证**:
- 手动 kill backend 进程后，health_monitor 在 20 秒内检测到并自动重启
- 对比之前需要 35 分钟（watchdog 默认间隔）才能恢复

### 1.3 async_audit_writer.py 线程本地存储修复

**问题**: 异步审计写入时，worker 线程访问主线程创建的 SQLite 连接，导致 `SQLite objects created in a thread can only be used in that same thread` 异常。

**修复**:
- 新增 `_tls = threading.local()` 线程本地存储
- 每个 worker 线程创建独立的 SQLite 连接
- 新增 `_ThreadLocalDS` 适配器类封装连接操作

**代码片段**:
```python
def _get_thread_ds(self):
    tls_ds = getattr(self._tls, 'ds', None)
    if tls_ds is not None:
        return tls_ds
    if self._ds is not None:
        # 为当前线程创建独立连接
        tls_ds = _ThreadLocalDS(self._ds.database)
        self._tls.ds = tls_ds
        return tls_ds
    return None
```

**验证**:
- 用户、角色、用户组创建操作的审计日志现在能正确写入
- 不再出现跨线程 SQLite 访问异常

## 二、FR-010 审计日志重试机制

### 2.1 audit_retry_worker.py 实现

**功能**: 后台 daemon 线程，每 60 秒扫描 `audit_logs` 表中 `action='AUDIT_WRITE_FAILED' AND status='failed'` 的记录，尝试重建原始审计日志。

**设计**:
- 独立后台线程，不阻塞主服务
- 只扫描 `status='failed'` 的记录，避免重复重试
- 成功后 `UPDATE status='retried'`，保留历史记录
- 失败记录保留 `status='failed'`，下次继续重试

**限制**:
- 只能恢复对象级别（object_type/id/action/user）
- 字段级别（field_name/old_value/new_value）数据丢失
- 历史 867 条 AUDIT_WRITE_FAILED（22:50 前，无 extra_data）无法重建

**核心代码**:
```python
class AuditRetryWorker:
    def __init__(self, data_source, interval_sec: int = 60, batch_size: int = 100):
        self._ds = data_source
        self._interval = interval_sec
        self._batch_size = batch_size
        self._running = False
        self._stats = {
            'scanned': 0,
            'retried': 0,
            'success': 0,
            'failed': 0,
        }
    
    def _scan_and_retry(self):
        # 扫描失败的审计日志
        rows = self._ds.execute("""
            SELECT id, object_type, object_id, user_id, user_name,
                   ip_address, user_agent, extra_data, created_at
            FROM audit_logs
            WHERE action='AUDIT_WRITE_FAILED' AND status='failed'
            ORDER BY id ASC
            LIMIT ?
        """, (self._batch_size,)).fetchall()
        
        for row in rows:
            self._retry_one(...)
    
    def _retry_one(self, audit_id, obj_type, obj_id, ...):
        # 从 extra_data 提取原始审计信息
        extra_data = json.loads(extra_data_str) if extra_data_str else {}
        original_action = extra_data.get('original_action', 'UNKNOWN')
        
        # 重建审计记录
        retry_record = {
            'object_type': obj_type,
            'object_id': obj_id,
            'action': original_action,
            'status': 'retried',
            'extra_data': json.dumps({
                'retry_source': 'audit_retry_worker',
                'original_audit_id': audit_id,
            }),
            ...
        }
        
        self._ds.insert('audit_logs', retry_record)
        self._ds.execute("UPDATE audit_logs SET status='retried' WHERE id=?", (audit_id,))
```

## 三、问题排查与修复

### 3.1 审计日志缺失问题

**现象**: 用户、角色、用户组创建操作无审计日志，其他操作有日志。

**根因**: 异步审计写入时，worker 线程访问主线程创建的 SQLite 连接，导致跨线程异常。

**修复**: 
- `async_audit_writer.py` 引入线程本地存储
- 每个 worker 线程创建独立 SQLite 连接
- 详见 L4 1.3 节

### 3.2 服务静默崩溃问题

**现象**: 服务崩溃后无法自动恢复，需要人工干预。

**修复**:
- `health_monitor.ps1` 每 5 秒检查 `/health` 端点
- 连续失败 2 次自动重启服务
- 详见 L4 1.2 节

### 3.3 服务状态不一致问题

**现象**: `service_manager.ps1 status` 报告的 PID 与实际监听进程 PID 不一致。

**修复**:
- 添加 listener-PID 校验
- 通过 `Get-NetTCPConnection` 获取实际监听进程 PID
- 详见 L4 1.1 节
