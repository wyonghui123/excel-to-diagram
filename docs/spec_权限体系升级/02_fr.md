# 3. 功能需求 (FR-001 ~ FR-015)

## FR-001: 权限从 BO actions 自动同步
- **描述**: 系统MUST在BO Schema注册时，自动从 `actions` 和 `category_config` 推导权限编码并写入 `permissions` 表。
- **验收标准**:
  - [AC-001.1] 新增BO YAML后，`permissions` 表自动出现对应权限记录（如 `domain:create`, `domain:read`, ...）
  - [AC-001.2] 修改BO的 action name 后，`permissions.name` 自动更新
  - [AC-001.3] `init_auth.py` 的 `seed_permissions()` 不再硬编码资源列表，改为调用 `PermissionSyncService`
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 竞品分析 + 当前断裂点#1/#2/#6/#7

## FR-002: 菜单 BO 元数据化
- **描述**: 系统MUST将 `menu_permissions` 表升级为完整的 YAML Schema 驱动的BO，菜单通过 YAML 定义，`required_permissions` 从关联的BO actions自动推导。
- **验收标准**:
  - [AC-002.1] 存在 `meta/schemas/menu.yaml` 作为菜单 BO 的唯一事实源
  - [AC-002.2] 菜单新增 `color`, `description`, `page_type`, `primary_object_type`, `object_types`, `auto_generated` 字段
  - [AC-002.3] `init_menu_permissions.py` 不再硬编码 `required_permissions` 和菜单定义
  - [AC-002.4] `MenuAutoGenerator` 能正确从 `MetaObject.actions` 推导 `required_permissions`
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 断裂点#3 + 单一事实源分析

## FR-003: 角色维度范围声明
- **描述**: 系统MUST支持在角色定义中声明管理维度范围（`role_dimension_scopes`），作为权限推导的入口。
- **验收标准**:
  - [AC-003.1] 存在 `role_dimension_scopes` 表（含 dimension_code, dimension_values, inherit_children, scope_mode）
  - [AC-003.2] 前端 RolePermissionCenter 新增"维度范围配置"面板（优先于当前的菜单勾选面板）
  - [AC-003.3] 维度值选择支持级联过滤（选 product 后 version 自动过滤）
  - [AC-003.4] `inherit_children=true` 时系统自动计算所有子级ID
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: SAP/用友BIP组织级别 + 用户讨论

## FR-004: 维度范围自动推导数据权限
- **描述**: 系统MUST从 `role_dimension_scopes` 自动生成 `permission_rules` 的条件规则。
- **验收标准**:
  - [AC-004.1] 声明 `version=[3], domain=[1,2,5]` 后, 自动生成 condition `"version_id=3 AND domain_id IN (1,2,5,子域ID)"`
  - [AC-004.2] 为所有适用的 resource_type 自动生成对应的 permission_rules 记录
  - [AC-004.3] 处理 `inherit_children` → 自动扩展子域ID
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 用友BIP主隔离维度 + 维度驱动方案

## FR-005: 维度范围自动推荐菜单
- **描述**: 系统MUST从维度范围推导该角色应该看到哪些菜单。
- **验收标准**:
  - [AC-005.1] 声明 `domain=[1,2,5]` 后, 自动推荐"领域管理""子领域管理""架构数据管理"
  - [AC-005.2] 菜单推荐逻辑: 菜单关联的BO在维度范围内有数据 → 推荐
  - [AC-005.3] 管理员可在推荐基础上增减菜单
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 维度驱动方案

## FR-006: 维度范围自动推导功能权限
- **描述**: 系统MUST从推荐菜单的 `required_permissions` + BO的 `category_config` 自动生成 `role_permissions` 记录。
- **验收标准**:
  - [AC-006.1] 推荐菜单后, `role_permissions` 自动包含菜单关联的所有功能权限
  - [AC-006.2] 管理员可微调（增减权限）
  - [AC-006.3] 调整菜单后权限自动联动更新
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 维度驱动方案 + SAP SU24风格

## FR-007: 10处硬编码映射消除
- **描述**: 系统MUST消除所有违反单一事实源原则的硬编码映射字典。
- **验收标准**:
  - [AC-007.1] `role_menu_api.py` 移除 `PERMISSION_LABELS` 和 `MENU_DISPLAY_NAMES` 字典
  - [AC-007.2] `init_auth.py` 移除 `_resource_name()` 和 `_action_name()` 函数
  - [AC-007.3] `data_permission_api.py` 的 `_get_resource_name()` 改用 MetaRegistry
  - [AC-007.4] 前端移除 `menuIconMap` 和 `getDefaultMenus()` 硬编码
  - [AC-007.5] `menuConfig.js` 废弃，改用 API 驱动
  - [AC-007.6] `MetaAction.ACTION_SUFFIX_MAP` 成为唯一的 action_id→suffix 映射（5行）
  - [AC-007.7] 方案中 `action_suffix_map` 只在 MetaAction 定义一次
  - [AC-007.8] `_generate_permission_name` 使用 `meta_obj.get_permission_label()`
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 单一事实源补充分析

## FR-008: 前端菜单完全由后端驱动
- **描述**: 前端MUST从后端 API 获取菜单，不再维护静态菜单配置。
- **验收标准**:
  - [AC-008.1] 存在 `GET /api/v1/menu/visible` 端点，返回当前用户可见的菜单树
  - [AC-008.2] 菜单数据包含: menu_code, menu_name, menu_path, icon, color, description, page_type, primary_object_type, sort_order, children
  - [AC-008.3] `menuConfig.js` 标记为 deprecated
  - [AC-008.4] `useMenuPermissions.js` 移除 `menuIconMap` 和 `getDefaultMenus()`
  - [AC-008.5] 路由守卫根据菜单权限动态判断页面可访问性（非仅 isAdmin）
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 断裂点#4/#5/#6

## FR-009: BO 字段级数据权限声明化
- **描述**: 系统MUST支持在BO YAML中声明 `data_permission_dimensions`。
- **验收标准**:
  - [AC-009.1] `MetaObject` 增加 `data_permission_dimensions` 字段
  - [AC-009.2] 支持 dimension 类型: owner_scope, context_scope, field_filter, organizational_scope
  - [AC-009.3] ImpactPreview 能根据 dimensions 自动展示影响的字段和范围
- **优先级**: Should
- **类型**: 功能性需求
- **来源**: SAP CAP @restrict + Mendix Access Rules

## FR-010: 字段级读写权限控制
- **描述**: 系统MUST激活 `PermissionAnnotation`，BOFramework 在 `get_ui_config()` 时叠加字段级 visible/editable。
- **验收标准**:
  - [AC-010.1] BO YAML 字段的 `permission.readable/writable/roles` 生效
  - [AC-010.2] edit 模式下，`permission.writable=false` 的字段显示为只读
  - [AC-010.3] 不满足 `permission.roles` 的字段不可见
- **优先级**: Should
- **类型**: 功能性需求
- **来源**: 细化方案方案5

## FR-011: 派生角色支持
- **描述**: 系统MUST支持同一功能角色×不同维度范围的派生角色。
- **验收标准**:
  - [AC-011.1] 角色可声明"维度可覆盖"标记
  - [AC-011.2] 基于同一角色 + 不同维度值创建派生角色
  - [AC-011.3] 派生角色自动继承父角色的功能权限模板
- **优先级**: Could
- **类型**: 功能性需求
- **来源**: SAP Derived Role + 用友BIP

## FR-012: ManagementDimension 升级为维度树
- **描述**: ManagementDimension MUST支持多棵维度树（借鉴用友BIP多维组织树）。
- **验收标准**:
  - [AC-012.1] management_dimensions 增加 tree_code, tree_level 字段
  - [AC-012.2] 同一实体可属于多棵维度树
  - [AC-012.3] DimensionScopeEngine 使用维度树替代硬编码的 hierarchy_chain
- **优先级**: Could
- **类型**: 功能性需求
- **来源**: 用友BIP多维组织树

## FR-013: actions 增加 action_group 分组
- **描述**: BO YAML 的 actions MUST支持 `action_group` 分组（如 standard/readonly/maintain）。
- **验收标准**:
  - [AC-013.1] actions 增加 `action_group` 字段
  - [AC-013.2] 角色配置支持按 action_group 分配（如分配"只读组"）
  - [AC-013.3] 前端按钮与 action_group 权限联动
- **优先级**: Could
- **类型**: 功能性需求
- **来源**: 用友BIP业务活动

## FR-014: 权限分析中心
- **描述**: 系统MUST提供用户权限全景视图和智能诊断功能。
- **验收标准**:
  - [AC-014.1] 可查询任一用户的完整权限清单（功能+菜单+数据）
  - [AC-014.2] 支持权限诊断：给定"资源+操作"，判断用户是否有权
  - [AC-014.3] 诊断结果包含建议（如"可能由缓存导致"）
- **优先级**: Could
- **类型**: 功能性需求
- **来源**: 用友BIP权限分析中心

## FR-015: 菜单权限与维度配置双模式共存
- **描述**: 系统MUST同时支持维度驱动模式和菜单驱动模式，共享底层表。
- **验收标准**:
  - [AC-015.1] 维度驱动和菜单驱动写同一套 role_permissions/role_menu_permissions/permission_rules 表
  - [AC-015.2] 管理员可在两种模式间切换
  - [AC-015.3] 现有角色不受影响
- **优先级**: Must
- **类型**: 功能性需求
- **来源**: 系统集成分析
