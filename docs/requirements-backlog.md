## 文档说明

本文档按照 MoSCoW 优先级方法组织需求，结合 PMBOK 项目管理方法论，为产品迭代提供清晰的优先级指引。

**优先级定义**：

| 优先级 | 标识 | 含义 | 交付要求 |
|--------|------|------|----------|
| Must Have | P0 | 核心功能，缺失则产品不可用 | 必须交付 |
| Should Have | P1 | 重要功能，提升产品价值 | 尽量交付 |
| Could Have | P2 | 优化功能，增强用户体验 | 视资源情况 |
| Won't Have | P3 | 暂不安排，当前版本不实现 | 未来考虑 |

**状态定义**：

| 状态 | 标识 | 含义 |
|------|------|------|
| 待开发 | 📋 | 已规划，尚未开始 |
| 开发中 | 🔄 | 正在开发 |
| 已完成 | ✅ | 开发完成 |
| 阻塞中 | 🚫 | 等待外部条件 |

## 10. 产品经理场景优化

### 10.1 中心范围+关系范围二级选择模式

**核心场景**：产品经理需要清晰了解"我负责的范围"与"这些范围与其他模块的关系"

| 功能点 | 描述 | 优先级 | 状态 | 备注 |
|--------|------|--------|------|------|
| **中心范围选择** | 支持选择业务对象集合作为中心范围，基于此展示业务对象/服务模块列表 | P1 | ✅ | CenterScopeSelector组件 |
| **关系结构树形展示** | 基于中心范围，展示关系统分类树（逐级展开） | P1 | ✅ | RelationCategoryTree组件 |
| **中心范围内关系选择** | 选择中心范围内的业务对象关系，按颗粒度分类 | P1 | ✅ | |
| **中心与外部关系选择** | 选择中心范围与外部的关系，按颗粒度分类 | P1 | ✅ | |
| **关系数量统计** | 每个层级节点显示关系数量统计 | P1 | ✅ | |
| **批量选择** | 支持按层级批量选择关系（如同选所有"跨领域"） | P2 | 📋 | 提升操作效率 |
| **搜索快速定位** | 支持按领域/子领域/服务模块/业务对象名称搜索定位关系 | P2 | 📋 | 大数据量场景优化 |
| **选中优先排序** | 选中的关系/节点排在前面展示 | P1 | ✅ | 用户体验优化 |

**关系分类结构**：

```
## 中心范围内对象关系（业务对象之间）
├── 跨领域 (n)
│   ├── 领域A - 领域B (n)
│   │   ├── 子领域A1 - 子领域B1 (n)
│   │   │   └── 服务模块... - 服务模块... (n)
├── 同领域跨子领域 (n)
│   ├── 领域A (n)
│   │   └── 子领域A1 - 子领域A2 (n)
├── 同子领域跨服务模块 (n)
│   └── 子领域A (n)
│       └── 服务模块1 - 服务模块2 (n)
└── 同服务模块 (n)
    └── 服务模块A (n)
        └── 业务对象... - 业务对象... (n)

## 中心范围与外部对象关系
├── 跨领域 (n)
│   └── 中心领域 - 外部领域X (n)
├── 同领域跨子领域 (n)
│   └── 中心子领域 - 外部子领域 (n)
├── 同子领域跨服务模块 (n)
│   └── 中心服务模块 - 外部服务模块 (n)
└── 同服务模块 (n)
    └── 中心业务对象 - 外部业务对象 (n)
```

```yaml
    type: string
    description: "open|acknowledged|resolved"
  - id: assigned_to
    name: 处理人
    type: string
```

#### Transactional Outbox 模式

```
业务操作 → 写入业务数据 + 写入 outbox 表 → 同一事务原子提交
                                              ↓
                              Outbox Reader (后台线程) → 投递到 WebSocket/Webhook
```

复用已有 `change_events` 表作为 outbox，扩展支持运营事件类型。

### 16.4 架构独立性分析

#### 核心问题：观察者的可观察性

#### 独立认证机制

```python
# 运维系统使用 admin_token 认证，不依赖用户表
ADMIN_TOKEN = os.environ.get('OPS_ADMIN_TOKEN')

def ops_auth_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        token = request.headers.get('Authorization', '').replace('Bearer ', '')
        if not token or token != ADMIN_TOKEN:
            return jsonify({'error': 'Unauthorized'}), 401
        return f(*args, **kwargs)
    return decorated
```

## 17. 数据库迁移方案 - TiDB HTAP

### 17.1 迁移背景与目标

**当前状态**：
- 数据库：SQLite (`architecture.db`)
- 数据源抽象层：已实现 `DataSource` 接口，支持 SQLite/MySQL/PostgreSQL
- 元模型驱动：26个 YAML 定义的业务对象
- 无存储过程依赖：业务逻辑在 Python 层

**迁移目标**：
- 目标数据库：TiDB（开源 HTAP 数据库）
- 核心价值：水平扩展 + HTAP 实时分析 + 金融级高可用

### 17.2 架构兼容性分析

#### 有利因素 ✅

| 因素 | 说明 | 影响 |
|------|------|------|
| **数据源抽象层** | `DataSource` 接口已实现，只需新增 TiDBAdapter | 代码改动小 |
| **元模型驱动** | Schema 自动生成，无需手写 DDL | 自动适配新数据库 |
| **无存储过程** | 业务逻辑在 Python 层 | TiDB 不支持存储过程，无影响 |
| **标准 SQL** | 使用标准 CRUD 操作 | 高度兼容 |
| **MySQL 适配器** | 已有 MySQLAdapter 可复用 | TiDB 兼容 MySQL 协议 |

#### 需要关注 ⚠️

| 风险项 | 说明 | 应对措施 |
|--------|------|---------|
| 自增主键 | TiDB 自增ID不连续 | 使用 `AUTO_ID_CACHE=1` 或业务生成ID |
| 热点问题 | 自增主键可能导致写入热点 | 使用 `SHARD_ROW_ID_BITS` 打散 |
| 大事务限制 | TiDB 限制事务大小 | 控制批量操作大小 |
| 运维复杂度 | 分布式数据库运维要求高 | 学习培训 |

### 17.5 TiDB 适配器实现要点

```python
# meta/core/sql_adapters.py - TiDBAdapter 复用 MySQLAdapter

class TiDBAdapter(MySQLAdapter):
    """TiDB 数据源适配器
    
    TiDB 高度兼容 MySQL 协议，可复用 MySQLAdapter，
    仅需调整以下差异：
    - 自增ID策略
    - 热点打散配置
    - 事务大小限制
    """
    
    @property
    def source_type(self) -> DataSourceType:
        return DataSourceType.TIDB
    
    @property
    def dialect(self) -> str:
        return "tidb"
    
    def create_table(self, table_name: str, columns: Dict[str, Dict], **options) -> bool:
        """创建表 - 添加 TiDB 特定优化"""
        # 添加热点打散配置
        shard_bits = options.pop("shard_row_id_bits", 4)
        result = super().create_table(table_name, columns, **options)
        if result and shard_bits:
            self.execute(
                f"ALTER TABLE {table_name} SHARD_ROW_ID_BITS = {shard_bits}"
            )
        return result
```

### 17.8 决策建议

**推荐迁移的前提条件**：

| 条件 | 当前状态 | 建议 |
|------|---------|------|
| 数据量 > 1TB | SQLite < 1GB | ⚠️ 暂不需要 |
| 高并发需求 | 单机应用 | ⚠️ 评估需求 |
| 实时分析需求 | 已有分析模型 | ✅ HTAP 有价值 |
| 高可用要求 | 无容灾 | ✅ 金融级高可用 |
| 运维能力 | 需学习 | ⚠️ 培训投入 |

**结论**：

- **当前 SQLite 能满足需求** → 暂缓迁移
- **预期业务增长/需要高可用/实时分析** → 推荐迁移
- **迁移成本低**（架构优势明显），风险可控
