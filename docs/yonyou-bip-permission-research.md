## 1. 用友BIP权限模型全景

### 1.1 总体架构

用友BIP的权限体系采用**RBAC + 组织隔离 + 数据权限**的三层架构：

```
┌──────────────────────────────────────────────────────────────┐
│  用友BIP 权限体系                                              │
│                                                              │
│  Step 0: 多维组织建模（权限的基础底座）                         │
│    ├── 行政组织树（刚性、稳定）                                 │
│    ├── 销售组织树（职能型）                                    │
│    ├── 会计主体树（财务视角）                                  │
│    ├── 人力树（HR视角）                                       │
│    ├── 审批汇报树（流程视角）                                   │
│    └── 每种组织树下可建多棵                                     │
│                                                              │
│  Step 1: 角色（RBAC核心）                                     │
│    ├── 功能权限：分配菜单 + 按钮级                          │
│    └── 组织权限：指定角色在哪些主组织范围内操作                  │
│                                                              │
│  Step 2: 授权（绑定用户与角色）                                │
│    ├── 按用户授权：选用户 → 分配角色 + 指定组织范围             │
│    └── 按角色授权：选角色 → 分配用户 + 指定组织范围             │
│                                                              │
│  Step 3: 数据权限（扩展的细粒度控制）                          │
│    ├── 主组织数据权限（组织隔离维度）                           │
│    ├── 员工数据权限（按人员过滤）                               │
│    ├── 用户数据权限（按登录用户过滤）                           │
│    └── 档案数据权限（按客户/供应商/物料等档案过滤）             │
│                                                              │
│  Step 4: 权限分析中心（运维保障）                              │
│    ├── 全景式权限总览（一键查看用户全部权限）                    │
│    ├── 智能诊断（定位权限问题 + 给出建议）                      │
│    └── 全员应用（普通用户可自查权限）                           │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 三大核心特性

| 特性 | 说明 | 对我们的参考价值 |
|------|------|----------------|
| **主隔离维度** | 多组织模式下，所有操作必须在指定组织范围内，组织是权限的第一前提 | ⭐⭐⭐⭐⭐ 与我们"维度驱动"理念完全一致 |
| **按钮级授权** | "管到指尖"——从菜单级下沉到按钮级，可控制单个按钮的可见性 | ⭐⭐⭐⭐ 我们的 actions 可以下沉到这个粒度 |
| **多维组织树** | 同一实体下多套组织树共存：行政+销售+财务+人力+审批 | ⭐⭐⭐⭐⭐ 我们的 ManagementDimension 可以借鉴多树模型 |

## 2. 主隔离维度 — 与我们"维度驱动"方案的直接验证

### 2.1 用友BIP的"主隔离维度"机制

> *"用户所有的操作权限都必须在有相应的主组织范围内，即在分配功能权限的同时需要指定用户可以在哪些主组织范围内进行业务处理。"*
>
> — 用友BIP权限管理官方文档

这是用友BIP权限体系最核心的设计——**组织是权限隔离的基本维度**：

```
┌──────────────────────────────────────┐
│ 用友BIP"主隔离维度"机制               │
│                                      │
│ 用户获得角色后，同时必须指定：          │
│   · 功能权限：能用哪些菜单/按钮        │
│   + 组织范围：在哪些组织内能用          │
│                                      │
│ 只有「功能权限 ∩ 组织范围」同时满足时  │
│ 用户才能进行业务操作                   │
│                                      │
│ 例如：                               │
│   用户：张三                          │
│   角色：采购员                        │
│   组织范围：华东区                     │
│                                      │
│  → 张三可以执行"采购员"功能            │
│  → 但只能操作"华东区"的数据            │
│  → 华南区的采购数据对张三是不可见的     │
└──────────────────────────────────────┘
```

### 2.2 与我们"管理维度为入口"方案的对应

| 用友BIP概念 | 我们的对应 | 状态 |
|------------|-----------|------|
| 主隔离维度（组织） | ManagementDimension（planned: role_dimension_scopes） | 🟡 我们已有 management_dimensions 表和 ConditionRuleDialog，但尚未成为角色配置的入口 |
| 角色 + 组织范围 | role + role_dimension_scopes | 🔴 我们方案中设计了这个表，尚未实现 |
| 功能权限+组织权限 | BO actions + dimension_scopes | 🟡 actions 已有，维度范围待实现 |
| 派生角色 | 同一角色 × 不同 dimension 值 | 🔴 我们方案中提到了这个设计 |

**验证结论**：用友BIP的主隔离维度机制，与我们提出的"管理维度为入口→维度范围声明→自动推导菜单+权限+数据"完全同构。这证明了我们的方案设计方向与国内头部企业软件的最佳实践一致。

## 3. 按钮级权限（Business Activity）— 粒度下沉的参考

### 3.1 用友BIP按钮级授权机制

用友BIP提供了从菜单级到按钮级的权限粒度下沉：

```
功能权限层级：
  菜单（Menu/Node）           ← 控制"能否看到这个页面"
    └── 业务活动（Business Activity） ← 控制"能用哪些按钮组合"
          └── 按钮（Button）          ← 如"新增"、"修改"、"删除"、"导出"、"打印"
```

**YonBIP旗舰版**的方式更直观：在【角色管理】中分配功能时，直接勾选每个菜单节点下的具体按钮：

```
角色：普通业务员
  ├── 菜单：客户管理   ☑
  │     ├── 查看      ☑
  │     ├── 导出      ☑
  │     ├── 新增      ☐
  │     ├── 修改      ☐
  │     └── 删除      ☐
```

**YonBIP高级版**引入了更抽象的概念——"业务活动"（Business Activity）：

```
注册一个"应用"节点（如"客户-业务单元"）
  → 为该应用注册多个"业务活动"：
      业务活动A（标准）：含所有按钮 [新增,修改,删除,查看,导出]
      业务活动B（只读）：含 [查看,导出]
      业务活动C（维护）：含 [新增,修改,查看]
  → 在角色中分配具体的业务活动而非整个应用
```

### 3.2 与我们的BO Actions的对照

| 用友BIP | 我们的架构 | 提升建议 |
|---------|-----------|---------|
| 菜单节点 | Menu | 已有方案，待实施 |
| 业务活动 | BO actions 中的业务类型 action | actions 可以进一步分类 |
| 按钮 | BO actions 中的 crud + 业务操作 | actions 已经有了，但前端按钮的可见性需要与 action 权限联动 |

**可借鉴的设计**：用友的"业务活动"概念相当于将 BO actions 分组打包。我们可以在 YAML 的 `actions` 中增加 `action_group` 字段：

```yaml
# domain.yaml
actions:
  - id: crud_create
    name: 创建领域
    type: crud
    action_group: standard      # ← 新增：所属的业务活动组
  - id: crud_read
    name: 查询领域
    type: crud
    action_group: readonly      # ← 只读组
  - id: crud_update
    name: 更新领域
    type: crud
    action_group: maintain      # ← 维护组
```

这样在分配角色时，可以直接分配"只读组"、"标准组"、"维护组"等 action_group，而不是逐个 action 勾选。

## 4. 多维组织树 — 对我们 ManagementDimension 的深化

### 4.1 用友BIP的多维组织模型

用友BIP最创新的组织设计是"多维组织树"：

```
业务单元（核心实体）
    ├── 行政组织树（法人视角）
    │     ├── 集团 → 子公司 → 事业部 → 部门
    │     └── 用于：汇报关系、行政审批
    │
    ├── 财务组织树（会计视角）  
    │     ├── 法人实体 → 利润中心 → 成本中心
    │     └── 用于：合并报表、成本核算
    │
    ├── 销售组织树（业务视角）
    │     ├── 销售大区 → 销售办事处 → 销售组
    │     └── 用于：销售业绩归集、客户分配
    │
    └── 人力组织树（HR视角）
          ├── 集团HR → 区域HR → 部门HR
          └── 用于：编制管理、人事审批
```

**关键机制**：
- 同一个业务单元，在不同组织树下可以有不同的父子关系和"汇报路径"
- 权限可以按照**不同的组织树**来分配，满足不同管理视角的权限需求
- 审批流可以按特定的组织树进行逐级审批
- 数据上报/下发可以沿组织树传递

### 4.2 与我们 ManagementDimension 的对照

| 用友BIP | 我们的架构 | 差距 |
|---------|-----------|------|
| 多维组织树 | ManagementDimension（单维度） | 🟡 我们目前是"一个维度一条记录"，没有组织树的概念 |
| 组织树下的父子关系 | cascade_parent（简单父子） | 🟡 我们的 cascade_parent 是一条线，不是一棵树 |
| 按组织树分配权限 | role_dimension_scopes（待实现） | 🔴 尚未实现 |
| 组织树作为审批路径 | 无 | 🔴 新需求 |

**可借鉴的设计**：我们的 ManagementDimension 可以升级为"维度树"模型：

```yaml
# management_dimension.yaml 新增字段
fields:
  - id: tree_code
    name: 所属维度树
    type: string
    description: 同一实体可能属于多棵维度树，如 domain 可在"业务维度树"和"技术架构树"中
    example: "business_tree" / "tech_architecture_tree"

  - id: tree_parent_code
    name: 树中的父节点编码
    type: string
    description: 维度树中的父节点（非数据库外键，而是维度间的逻辑层级）

  - id: tree_level
    name: 树层级
    type: integer
    description: 在维度树中的层级深度
```

这样 product → version → domain → sub_domain → service_module → business_object 就不再是硬编码的 hierarchy_chain，而是可以从 management_dimensions 表动态读取的**可配置的维度树**。

## 5. 用友BIP的元数据驱动架构与我们的对比

### 5.1 用友BIP的元数据驱动

用友BIP的架构文档明确声明其核心架构为 **"云原生(含微服务)、元数据驱动、中台化和数用分离"**。

其在低代码平台 YonBuilder 中的元数据驱动流程：

```
元数据建模（实体 + 字段 + 引用关系）
    ↓
页面建模（自动生成列表页 + 详情页 + 表单）
    ↓
权限注入（元数据标签：如"数据权限控制"标签）
    ↓
应用发布
```

**与我们高度一致的路径**：实体=BO，字段=fields，引用关系=relations，页面建模=ui_view_config，权限注入=category_config + actions。

### 5.2 数据权限的元数据声明式

用友BIP的低代码数据权限采用元数据标签机制：

```yaml
# 用友BIP方式（伪代码表示）
entity: 采购订单
  fields:
    - field: org_id
      reference: "主组织"       # ← 标签化：标记为组织维度字段
      tags: [数据权限控制]
    
    - field: create_user
      reference: "员工参照"
      tags: [数据权限控制]       # ← 标记为员工权限控制字段

# 页面设计器：
#   ☑ 开启数据权限控制          # ← 一键开关

# 运行时：
#   自动注入 org_id IN (用户授权的组织范围)
```

与我们 `data_permission_dimensions` 方案的对应：

| 用友BIP | 我们的方案 |
|---------|-----------|
| `字段添加标签"数据权限控制"` | `data_permission_dimensions: [{field: org_id, type: org_scope}]` |
| `列表设计器开启数据权限控制` | BOFramework 的 `get_ui_config()` 自动叠加权限 |
| `"授权"菜单分配组织权限` | `role_dimension_scopes` + DimensionScopeEngine |

## 6. 权限分析中心 — 运维能力参考

### 6.1 用友BIP权限分析中心

这是用友BIP的一个创新功能，提供：

1. **全景式权限总览**：一键查看任一用户的完整权限清单（功能权限 + 组织权限）
2. **智能诊断**：输入"服务 + 活动"，系统自动判断该用户是否有权，并返回诊断建议
3. **诊断建议**：给出针对性的解决方案（如"可能由缓存导致，建议清理缓存"）
4. **全员应用**：普通用户可以自查权限，减少IT支持工作量

### 6.2 与我们的RolePermissionCenter对比

| 功能 | 用友BIP | 我们当前的RolePermissionCenter |
|------|---------|-------------------------------|
| 角色权限矩阵 | ✅ 有 | ✅ 有（MenuPermissionMatrix） |
| 影响范围预览 | ❓ 未知 | ✅ 有（ImpactPreview，我们独有的） |
| 用户权限总览 | ✅ 全景式一览 | 🟡 部分（只能看角色维度） |
| 智能诊断 | ✅ 含建议 | ❌ 无 |
| 用户自查 | ✅ 支持 | ❌ 无 |

**可借鉴**：为我们的 `RolePermissionCenter` 增加"用户权限总览"和"权限智能诊断"功能。

## 9. 总结

用友BIP作为国内企业服务领域的头部产品，其权限模型与我们方案的高度一致性令人鼓舞：

1. **"组织是权限隔离的基本维度"** — 用友BIP的核心设计原则，与我们"维度驱动"方案完全一致
2. **"主隔离维度"** — 与我们 `role_dimension_scopes` 的设计理念一致，且用友已在大规模企业客户中验证
3. **"多维组织树"** — 超越了我们当前的线性 hierarchy，为我们 ManagementDimension 的深化提供了方向
4. **"业务活动(按钮级权限)"** — 为我们 actions 的 `action_group` 设计提供了参考
