分组模型统一渲染方案

核心命题：一切皆分组 → 容器嵌套容器 → 叶子即节点 → 统一渲染

一、核心命题验证：图表能否统一为"嵌套容器+节点"？

1.1 当前两种图的渲染本质

BO 业务对象图渲染

  • Mermaid结构：subgraph(领域) → subgraph(子领域) → subgraph(服务模块) → 节点(业务对象)
  • 本质：3层嵌套容器 + 叶子节点
  • 当前代码：useBusinessObjectSyntax.js 约1073行，包含颜色计算、映射重建等

SM 服务模块图渲染

  • Mermaid结构：subgraph(领域) → subgraph(子领域) → 节点(服务模块)
  • 本质：2层嵌套容器 + 叶子节点
  • 当前代码：useServiceModuleSyntax.js 约437行，直接使用构建结果

结论：两种图的渲染本质完全相同——都是"嵌套容器+叶子节点"。

差异仅在于：嵌套深度不同（BO图3层容器，SM图2层容器）和叶子节点类型不同（BO图叶子是BusinessObject，SM图叶子是ServiceModule）。这恰好就是GroupModel已经抽象出来的概念——isTerminalGroup()判断是否为叶子，GroupType区分类型，children构建嵌套。

1.3 统一渲染的核心思路

统一渲染算法：遍历GroupModel树 → 容器/节点二分法
function renderGroupModel(groupModel, links) {
  const mermaidCode = 'flowchart LR\n'
  
  // 递归遍历 GroupModel 树
  function renderGroup(group, depth = 0) {
    const isTerminal = isTerminalGroup(group, chartType)
    
    if (isTerminal) {
      // 叶子 = 节点：渲染为矩形
      renderNode(group)
    } else {
      // 非叶子 = 容器：渲染为 subgraph
      renderContainer(group, depth)
    }
  }
  
  function renderNode(group) {
    // group.color — 从GroupModel读取（新增字段）
    // group.title — 显示名称
    // group.elementRef.code — 编码
    mermaidCode += `  ${group.id}["${group.title}\n(${group.elementRef.code})"]\n`
    mermaidCode += `  style ${group.id} ${getNodeStyle(group.color)}\n`
  }
  
  function renderContainer(group, depth) {
    mermaidCode += `  subgraph ${group.id}["${group.title}"]\n`
    mermaidCode += `    direction ${group.layout.direction}\n`
    
    // 递归渲染子分组
    group.children.forEach(child => renderGroup(child, depth + 1))
    
    mermaidCode += `  end\n`
    mermaidCode += `  style ${group.id} ${getContainerStyle(group.color)}\n`
  }
  
  // 渲染连线（叶子节点之间的连线）
  links.forEach(link => {
    mermaidCode += `  ${link.source} --> ${link.target}\n`
  })
  
  return mermaidCode
}
[WARNING] 关键洞察：当前useBusinessObjectSyntax.js的1073行中，约400行是在做"从diagramData重建分组映射"的工作——把nodes按domain/subDomain/serviceModule重新分组。如果直接从GroupModel树渲染，这400行代码完全不需要！GroupModel本身就是分组后的树结构，无需重建。

三、统一渲染管线的完整设计

3.1 新管线 vs 旧管线

旧管线（当前）

  • previewData → filteredData
  • filteredData → buildDiagramData() → diagramData
  • diagramData → useBusinessObjectSyntax() → 重建映射 → 计算颜色 → 生成Mermaid
  • 问题：渲染层需要重建分组映射（~400行），颜色双路径计算

新管线（统一）

  • previewData → filteredData
  • filteredData → GroupModel（增强color/isCenter）
  • GroupModel + links → UnifiedRenderer → 生成Mermaid
  • 优势：GroupModel就是分组后的树，无需重建；颜色在写入GroupModel时统一计算

3.2 新管线的阶段设计

统一渲染管线 — 5个阶段，仅1个分化点
Stage 1: 数据输入 + 过滤（共享，与当前相同）

3.3 UnifiedRenderer核心算法

UnifiedRenderer — 统一渲染器，替代两个Syntax文件
export class UnifiedRenderer {
  render(groupModel, links, options) {
    const { layoutEngine, chartType } = options
    const rootGroups = groupModel.getRootGroups()
    const nodeIdMap = buildNodeIdMap(rootGroups, chartType)
    
    let code = layoutEngine === 'elk' 
      ? 'flowchart-elk LR\n' 
      : 'flowchart LR\n'
    
    // 1. 递归渲染分组树
    rootGroups.forEach(group => {
      code += this.renderGroup(group, chartType, nodeIdMap, 0)
    })
    
    // 2. 渲染连线
    links.forEach(link => {
      const sourceId = nodeIdMap.codeToIdMap.get(link.source)
      const targetId = nodeIdMap.codeToIdMap.get(link.target)
      if (sourceId && targetId) {
        code += `  ${sourceId} --> ${targetId}\n`
      }
    })
    
    // 3. 渲染样式
    rootGroups.forEach(group => {
      code += this.renderStyles(group, chartType)
    })
    
    return code
  }
  
  renderGroup(group, chartType, nodeIdMap, depth) {
    const isTerminal = isTerminalGroup(group, chartType)
    
    if (isTerminal) {
      // 叶子 = 节点
      const displayText = group.elementRef.code 
        ? `${group.title}\\n(${group.elementRef.code})` 
        : group.title
      return `  ${group.id}["${displayText}"]\n`
    }
    
    // 非叶子 = 容器
    let code = `  subgraph ${group.id}["${group.title}"]\n`
    code += `    direction ${group.layout.direction}\n`
    
    group.children.forEach(child => {
      code += this.renderGroup(child, chartType, nodeIdMap, depth + 1)
    })
    
    code += `  end\n`
    return code
  }
  
  renderStyles(group, chartType) {
    const isTerminal = isTerminalGroup(group, chartType)
    let code = ''
    
    if (isTerminal) {
      code += `  style ${group.id} ${getNodeStyle(group.color, group.textColor)}\n`
    } else {
      code += `  style ${group.id} ${getContainerStyle(group.color)}\n`
    }
    
    group.children?.forEach(child => {
      code += this.renderStyles(child, chartType)
    })
    
    return code
  }
}

五、核心结论与实施建议

图表大方向完全可以统一成一个——通过分组模型完成统一。

两种图在渲染层面的本质完全相同：嵌套容器+叶子节点。GroupModel已经具备了统一渲染所需的大部分能力（树形结构、容器/节点区分、层级控制、启用/禁用），仅缺少颜色和标注信息。通过enrichGroupModel步骤补充这些信息后，一个UnifiedRenderer即可替代当前的两个独立Syntax文件（共1510行），代码量减少约69%。

统一后的渲染管线极简：

  • 输入：enrichedGroupModel（带颜色的分组树）+ links（连线数组）
  • 算法：递归遍历GroupModel树，isTerminal→渲染节点，!isTerminal→渲染容器subgraph
  • 输出：Mermaid/mxGraph代码

最大的收益：当前useBusinessObjectSyntax.js中约400行的"从diagramData重建分组映射"逻辑将完全不需要——因为GroupModel本身就是分组后的树结构。这是消除代码重复和一致性风险的根本途径。

实施建议：分两步走——

  • 第一步：实现enrichGroupModel + UnifiedRenderer，先用SM图验证（SM图更简单，2层嵌套）
  • 第二步：验证通过后，将BO图也切换到UnifiedRenderer，删除旧的diagramDataBuilder和两个Syntax文件
