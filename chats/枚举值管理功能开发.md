# 枚举值管理功能开发

## 对话时间
- 日期：2026-04-29
- 主要内容：枚举值管理功能的元模型设计、用户交互方案设计、实现和测试

---

## 一、需求背景

用户提出需要管理业务枚举值（如关系类型、备注类型等），需要支持：
1. 用户可维护的简单1维枚举
2. 多维枚举（不同产品的标识不同）
3. 不可维护的系统枚举

---

## 二、行业最佳实践研究

### 2.1 SAP One Domain Model
SAP 采用分层管理策略：
- **Domain Fixed Values**：在 Domain 层定义 Value Range，适用于系统级稳定枚举
- **Check Table / Value Table**：通过数据库表存储，通过 SM30 维护
- **Customizing Table**：复杂业务配置，支持传输请求

**核心结论**：SAP **没有**使用完整的 BO 模型来管理枚举值，而是用配置表 + 维护视图的轻量方案。

### 2.2 Salesforce Value Set
- Global Value Set：跨字段共享
- Standard Value Set：系统预定义
- Custom Value Set：用户可维护

### 2.3 推荐架构
```
三层枚举模型：
├── SystemEnum（系统枚举）
│   ├── 代码定义，不可修改
│   └── 升级覆盖
├── BusinessEnum（业务枚举）
│   ├── 用户可扩展
│   └── 版本控制
└── ValueSet（值集）
    ├── 可复用
    ├── 多字段共享
    └── 维度支持
```

## 三、元模型设计

### 3.1 枚举类型 (enum_type)
| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(64) | 枚举类型ID |
| name | VARCHAR(128) | 显示名称 |
| category | VARCHAR(32) | 分类：system / business |
| mutability | VARCHAR(32) | 可维护性：locked / extensible / fully_editable |
| dimension_schema | JSON | 维度定义 |
| description | TEXT | 描述 |

### 3.2 枚举值 (enum_value)
| 字段 | 类型 | 说明 |
|------|------|------|
| id | SERIAL | 技术主键 |
| enum_type_id | VARCHAR(64) | 所属枚举类型 |
| code | VARCHAR(64) | 编码 |
| name | VARCHAR(256) | 显示名称 |
| name_en | VARCHAR(256) | 英文名称 |
| dimensions | JSON | 维度值 |
| sort_order | INT | 排序 |
| is_active | BOOLEAN | 是否启用 |
| is_system | BOOLEAN | 是否系统预置 |
| parent_code | VARCHAR(64) | 父级编码（层级枚举） |

### 3.3 可维护性控制
| mutability | 含义 | 新增 | 编辑 | 删除系统值 | 删除用户值 |
|-----------|------|------|------|-----------|-----------|
| locked | 锁定 | ❌ | ❌ | ❌ | ❌ |
| extensible | 可扩展 | ✅ | ✅ | ❌ | ✅ |
| fully_editable | 完全可编辑 | ✅ | ✅ | ❌ | ✅ |

## 四、用户交互方案

### 4.1 枚举类型列表页
- 显示：编码、名称、分类、可维护性、维度数、值数量
- 支持筛选：分类、可维护性
- 支持搜索
- 新建业务枚举按钮

### 4.2 枚举值管理页
- 面包屑导航返回
- 维度标签页切换（多维枚举）
- 枚举值列表：编码、名称、排序、状态、来源
- 新增/编辑/删除功能
- 权限检查：系统值不可删除

### 4.3 字段配置集成
```python
MetaField(
    id="relation_type",
    name="关系类型",
    field_type=FieldType.STRING,
    enum_type="relation_type",
    enum_filter={"product": "ERP"}  # 维度过滤
)
```

### 5.3 数据库表

```sql
CREATE TABLE enum_types (
    id VARCHAR(64) PRIMARY KEY,
    name VARCHAR(128) NOT NULL,
    category VARCHAR(32) NOT NULL,
    mutability VARCHAR(32) NOT NULL,
    dimension_schema JSON,
    description TEXT,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

CREATE TABLE enum_values (
    id SERIAL PRIMARY KEY,
    enum_type_id VARCHAR(64) NOT NULL,
    code VARCHAR(64) NOT NULL,
    name VARCHAR(256) NOT NULL,
    name_en VARCHAR(256),
    dimensions JSON,
    sort_order INT DEFAULT 0,
    is_active BOOLEAN DEFAULT TRUE,
    is_system BOOLEAN DEFAULT FALSE,
    parent_code VARCHAR(64),
    metadata JSON,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    UNIQUE(enum_type_id, code),
    FOREIGN KEY (enum_type_id) REFERENCES enum_types(id)
);
```

## 七、关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 系统枚举 vs 业务枚举 | 分开管理，category 区分 | 系统枚举需要代码级控制 |
| 可维护性控制 | mutability 三级控制 | 满足不同场景需求 |
| 多维支持 | dimensions JSON 字段 | 灵活扩展，避免多表关联 |
| 值来源 | 静态值 + 动态值表 | 兼顾简单场景和复杂场景 |
| 国际化 | name + name_en | 支持多语言 |
| SAP 方案 | 配置表 + API | 轻量级，无需完整 BO 框架 |

## 八、使用示例

### 8.1 创建多维枚举类型
```bash
POST /api/v1/enum-types
{
    "id": "relation_type",
    "name": "关系类型",
    "category": "business",
    "mutability": "extensible",
    "dimension_schema": [{"id": "product", "name": "产品"}]
}
```

### 8.2 创建枚举值
```bash
POST /api/v1/enum-types/relation_type/values
{
    "code": "PARENT_CHILD",
    "name": "父子关系",
    "dimensions": {"product": "ERP"}
}
```

### 8.3 按维度过滤查询
```bash
GET /api/v1/enum-types/relation_type/values?product=ERP
```
