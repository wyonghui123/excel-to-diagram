# SM图表外部SM容器渲染修复 - 2026-06-11

## 问题描述

在架构管理页面，用户配置：
- **对象范围**: 供应链管理系统 / 新测试2_1780899784189，勾选 "采购管理 / 采购需求" 子领域下的 3 个 SM（采购需求、供应商管理、采购合同）
- **关系范围**: 范围内 (2) + 范围内与外部 (7)
- **图表配置**: 服务模块图 + 按服务模块颜色分组 + 中心范围高亮

**症状**：
- 配置阶段颜色分组正确显示了 5 个中心范围外部的 SM（采购执行、货位管理、入库管理、应付发票、付款计划）
- 图表展示阶段，**这 5 个外部 SM 没有渲染为 SM 容器**，其业务对象都变成了**蓝色的孤儿节点**

## 修复历程

### v31 修复（不完整）

在 [useDiagramData.js:1262-1317](src/views/AADiagramApp/composables/useDiagramData.js) 为 SM 图表的 `filteredDomainProducts` 添加了与 v29 BO 图表同款的 orphan hierarchy 补齐逻辑。

**问题**：v31 修复方向错误。
- `filteredDomainProducts` 是前端计算属性，会被 `buildDomainProducts` 影响
- 但图表实际渲染用的是 `smFromContainers`（来自 `filteredContainers`），而 `filteredContainers` 在 [useDiagramData.js:668-704](src/views/AADiagramApp/composables/useDiagramData.js) 直接遍历 `previewData.value.serviceModules`
- `previewData.value.serviceModules` 来自后端 API，**只包含用户勾选的 SM，外部 SM 完全缺失**

### v32 修复（根因解决）

**根因**：[bo_api.py:1054-1055](meta/api/bo_api.py) 严格按 `module_id_list` 过滤 `modules`：

```python
if module_id_list:
    modules = [m for m in modules if m.get('id') in module_id_list]
```

但同时 `business_objects` 在 `bo_id_list` 为空时**不过滤**（line 1056-1057）。
- 用户在 SM 层勾选时，`bo_id_list` 为空 → `business_objects` 包含所有 SM 的 BO
- 但 `modules` 只剩用户勾选的 3 个
- 前端 `buildServiceModules()` 遍历 `serviceModules` 不会为外部 SM 创建容器
- 外部 BO 因为没有对应 SM 容器 → 蓝色孤儿节点

**修复**：[bo_api.py:1060-1103](meta/api/bo_api.py)，在严格过滤后从 `business_objects` 反推缺失的 SM/SD/Domain 并补回：

## 核心经验

### 经验 1：Hierarchy 过滤的"过严"是数据孤儿的常见根因

**问题模式**：
- 后端 API 按父级 ID 严格过滤子级数据（如 `module_id_list` 过滤 `modules`）
- 但 BO/SM 关系是图状的，外部 BO 会引用未过滤范围内的 SM
- 严格过滤导致 metadata 缺失，但 BO 数据保留 → 孤儿节点

**修复原则**：
- 当 API 同时返回 `business_objects`（不过滤）和 `service_modules`（严格过滤）时，必须保证 **被任何 BO 引用的 SM 都在 modules 列表里**
- 类似地，被任何 sub_domain 引用的 domain 也要保留
- 简单规则：层级数据 = 范围 ID 列表 ∪ BO 反推的引用 ID 列表

### 经验 2：前端 orphan 补齐是兜底，不是根治

v29 (BO 图表) 和 v31 (SM 图表) 的 orphan 补齐逻辑（`filteredDomainProducts`）只解决了**前端聚合后**的层级问题，**没有解决源头数据缺失问题**：
- `previewData.value.serviceModules` 在源头就缺
- `filteredContainers` (SM 图表) 用 `previewData.value.serviceModules` → 容器不渲染
- 即使 `filteredDomainProducts` 补齐了，容器还是不出来

**判断方式**：如果"配置阶段显示正常"但"图表阶段不显示"，**根因在数据源头**，前端补齐无效。

### 经验 3：诊断顺序

遇到"外部 X 容器缺失，业务对象变孤儿"的问题：
1. **先用 API 直接验证**：用相同参数调 `architecture/preview`，看 `service_modules` 字段是否包含所有需要的 SM
2. **不要只看前端的 computed**：前端多个 computed 互相组合，一个修了另一个可能没修
3. **追踪数据流到源头**：从 `previewData` → API 调用 → 后端 SQL，找到第一个缺失数据的地方

### 经验 4：多层级 API 过滤要"保底扩展"

后端聚合 API（如 `architecture/preview`）一次性返回多层级数据时，过滤逻辑要**自下而上**而非**自上而下**：
- ❌ 自上而下：按 `module_id_list` 过滤 modules → 丢失外部 SM
- ✅ 自下而上：先收集所有 BO → 反推引用的 SM/SD/Domain → 合并到结果

这是图状数据的特性：父子关系不是严格 1:N，BO 可以"跨范围"引用 SM。
