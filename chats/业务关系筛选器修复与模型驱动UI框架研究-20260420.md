## 一、问题背景

### 1.1 业务场景

业务关系管理页面需要实现：
- 产品/版本自动选择
- 业务对象树形选择
- 关系范围（分类维度）树形选择
- 基于筛选条件的关系列表过滤

### 1.2 数据模型

```
关系范围树结构：
├── 中心范围内 (scope_type)
│   ├── 跨领域 (category_type)
│   │   └── 领域对 (pair/leaf)
│   ├── 同领域跨子领域 (category_type)
│   └── 同服务模块 (category_type)
└── 中心范围外 (scope_type)
    └── ...
```

### 1.3 核心问题

1. **树节点选中状态不同步**：父节点选择后子节点不会被选中
2. **关系列表为空**：点击"应用筛选"后右侧列表无数据
3. **过滤器选项不匹配**：relation_code 选项与数据库实际数据不一致

## 二、筛选器问题修复

### 2.1 问题根因

`selectedKeys` 数组存储的是 `category_type` 级别的 key（如 `cross_domain`），但叶子节点检查的是自己 pair 的 key，两者不匹配。

```
树结构 vs selectedKeys 的关系：
中心范围内 (scope_type)          ← 检查所有子 category_type 是否全选
  └─ 跨领域 (category_type)      ← key = "cross_domain"  ✓ 存在 selectedKeys 中
       ├─ 采购-采购 (pair/leaf)   ← key = "1-2"           ✗ 不在 selectedKeys 中！
       └─ 营销-供应链 (pair/leaf) ← key = "3-4"           ✗ 不在 selectedKeys 中！
```

### 2.2 修复方案

新增 `ancestorCategoryKey` prop，让每个节点知道自己所属的 category_type key：

**FilterTreeNode.vue 核心修改：**

```vue
<script setup>
const props = defineProps({
  node: { type: Object, required: true },
  selectedKeys: { type: Array, default: () => [] },
  expandedKeys: { type: Set, default: () => new Set() },
  level: { type: Number, default: 0 },
  showCount: { type: Boolean, default: true },
  filterKey: { type: String, default: '' },
  ancestorCategoryKey: { type: String, default: '' }  // 新增
})

function getNodeCategoryKey(node) {
  if (node.level === 'category_type') return node.key
  return props.ancestorCategoryKey
}

const currentCategoryKey = computed(() => getNodeCategoryKey(props.node))

const isSelected = computed(() => {
  if (props.filterKey === 'category_type') {
    const catKey = currentCategoryKey.value
    if (props.node.level === 'scope_type') {
      const childCatKeys = collectKeysByLevel(props.node, 'category_type')
      return childCatKeys.length > 0 && childCatKeys.every(k => props.selectedKeys.includes(k))
    }
    return catKey ? props.selectedKeys.includes(catKey) : false
  }
  // ... 其他逻辑
})
</script>

<template>
  <!-- 传递 ancestorCategoryKey 给子节点 -->
  <FilterTreeNode
    v-for="child in node.children"
    :key="child.id"
    :node="child"
    :ancestor-category-key="currentCategoryKey"
    ...
  />
</template>
```

### 2.3 关键文件修改

| 文件 | 修改内容 |
|------|---------|
| `FilterTreeNode.vue` | 添加 `ancestorCategoryKey` prop，修改 `isSelected`/`isIndeterminate` 计算逻辑 |
| `DynamicFilter.vue` | 传递 `filterKey` prop，修复 `selectAllTree` 函数 |
| `manage_api.py` | 将 `category_type` 过滤移到 SQL 层面 |

## 三、关系列表过滤问题

### 3.1 问题现象

前端请求参数正确，但后端返回 0 条数据：
```
useApi.list - URL: /api/v1/relationships?...&relation_codes=CALLS&relation_codes=DEPENDS_ON...
DynamicView - API result: true total: 0 data count: 0
```

### 3.2 根因分析

`relation_code` 过滤器的选项与数据库实际数据不匹配：

| YAML 配置的选项 | 数据库实际数据 |
|----------------|---------------|
| CALLS, DEPENDS_ON, DATA_FLOW, INCLUDE | PROVIDES, ORDERS, CONTAINS, RESERVES, GENERATES, SUPPLIES, ... |

### 3.3 修复内容

**1. 更新 relationship.yaml：**
```yaml
- key: relation_code
  title: 关系类型
  type: checkbox_group
  options:
    - value: PROVIDES
      label: 提供
    - value: ORDERS
      label: 订购
    - value: CONTAINS
      label: 包含
    # ... 更多实际存在的选项
  # 移除 default: all
```

**2. 修复 yaml_loader.py 默认值问题：**
```python
# 原来：default=data.get("default", "all")
# 修改为：default=data.get("default")  # 不设置默认值
```

**3. 修复 meta_api.py 默认值问题：**
```python
# 只在 YAML 明确设置时才返回 default
if hasattr(f, 'default') and f.default is not None:
    filter_item['default'] = f.default
```

### 3.4 后端 SQL 层面过滤优化

将 `category_type` 过滤从 Python 层面移到 SQL 层面：

```python
category_conditions = []
if category_types:
    for ct in category_types:
        if ct == 'cross_domain':
            category_conditions.append("sd1.domain_id != sd2.domain_id")
        elif ct == 'same_domain_cross_subdomain':
            category_conditions.append("sd1.domain_id = sd2.domain_id AND sd1.id != sd2.id")
        elif ct == 'same_subdomain_cross_module':
            category_conditions.append("sd1.id = sd2.id AND sm1.id != sm2.id")
        elif ct == 'same_module':
            category_conditions.append("sm1.id = sm2.id")
    
    if category_conditions:
        where_clause += " AND (" + " OR ".join(category_conditions) + ")"
```

## 四、模型驱动UI框架研究

### 4.1 行业主流框架

| 框架 | 技术栈 | GitHub Stars | 核心特点 |
|------|--------|--------------|----------|
| **Formily (阿里)** | React/Vue | 11k+ | MVVM架构、高性能、联动强大 |
| **FormKit** | Vue 3 | 4.6k+ | 单组件、Schema驱动、开箱即用 |
| **JSON Forms** | React/Vue/Angular | 2.6k+ | 标准JSON Schema、多框架 |
| **React Hook Form** | React | 41k+ | 高性能、非受控、API简洁 |
| **TanStack Form** | React/Vue/Solid | 3k+ | 框架无关、类型安全 |

### 4.2 Formily 架构

```

├─────────────────────────────────────────────────────────────────┤
│                        扩展组件层                                │
│  @formily/antd │ @formily/element │ @formily/next              │
├─────────────────────────────────────────────────────────────────┤
│                        UI 桥接层                                │
│  @formily/react │ @formily/vue                                 │
├─────────────────────────────────────────────────────────────────┤
│                        内核层                                   │
│  @formily/core │ @formily/reactive │ @formily/path             │
├─────────────────────────────────────────────────────────────────┤
│                        响应式引擎                               │
│  Observable │ Tracker │ Autorun │ Reaction                     │
└─────────────────────────────────────────────────────────────────┘
```

## 八、最终结论与建议

### 8.1 推荐策略

```
阶段 1: 立即 (0-1月) - 保持现状 + AI 优化
├── 使用 AI 快速修复现有问题
├── 使用 AI 优化性能瓶颈
└── 使用 AI 完善文档和测试

阶段 2: 短期 (1-3月) - 小范围尝试 Formily
├── 选择 1-2 个新功能使用 Formily
├── 使用 AI 辅助学习和开发
└── 建立 YAML → Formily Schema 转换

阶段 3: 中期 (3-6月) - 评估并决策
├── 评估 Formily 使用效果
└── 决定是否全面迁移

阶段 4: 长期 (6-12月) - 按需全面迁移
├── 如果效果好，使用 AI 辅助全面迁移
└── 如果效果一般，保持混合架构
```

### 8.2 核心结论

| 维度 | 结论 |
|------|------|
| **AI 改变了什么** | 大幅降低学习和迁移成本，ROI 从 6% 提升到 198% |
| **AI 对哪个方案更有利** | Formily (提升 +16 vs +9) |
| **当前推荐** | 保持现状 + AI 优化，同时小范围尝试 Formily |
| **长期推荐** | 如果验证成功，全面迁移到 Formily + AI |
| **关键成功因素** | AI 辅助 + 渐进式迁移 + 持续验证 |

### 8.3 决策依据

1. **当前需求已基本满足** - 80% 核心功能已实现
2. **AI 显著改变成本结构** - 迁移成本降低 64%
3. **AI 对 Formily 加持更大** - Formily 有大量公开文档和示例
4. **渐进式降低风险** - 先小范围验证，再决定是否全面迁移
