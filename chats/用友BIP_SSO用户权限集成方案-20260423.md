## 第一部分：SSO、用户与权限集成方案

### 一、方案概述

用户的应用部署在175内部服务器，需要与用友BIP系统集成，实现：
- **单点登录（SSO）**：用户通过用友友户通统一认证后，可免登录访问您的应用
- **用户同步**：从用友BIP同步用户信息到您的应用
- **权限管理**：基于用友BIP的角色和权限体系，控制用户在您应用中的访问权限

### 二、SSO单点登录集成方案

#### 2.1 推荐方案：集成认证中心

用友BIP提供了**集成认证中心**功能，支持三种单点登录协议：

| 协议 | 核心思想 | 适用场景 | 推荐度 |
|------|---------|---------|--------|
| **CAS** | 基于票据的代理认证 | 传统Web应用 | ⭐⭐⭐⭐⭐ |
| **OAuth 2.0** | 授权框架，借用来做认证 | 第三方应用授权，移动端 | ⭐⭐⭐⭐ |
| **SAML** | 基于XML的断言 | 企业级SSO（如与AD集成） | ⭐⭐⭐ |

**推荐使用CAS协议**，因为：
1. 适合传统Web应用
2. 用友BIP原生支持
3. 实现相对简单

#### 2.2 集成认证中心配置步骤

**步骤1：配置集成认证中心**
- 访问路径：`数字化建模 > 系统管理 > 集成认证中心`
- 前提条件：当前企业账号需完成企业认证
- 操作：点击"新增集成认证"，系统会分配一个唯一的编码（thirdUcId）

**步骤2：对接开放平台接口**
- 获取单点登录code的接口文档：https://open.diwork.com/#/doc-center/docDes/api?code=yonbip&section=d1c38bd20e2e41fbaf200c4fdf883d9b
- 需要先完成开放平台 access_token 鉴权

**步骤3：拼接登录地址**
```
https://euc.diwork.com/cas/thirdOauth2CodeLogin?thirdUcId={thirdUcId}&code={code}&service={service}
```
参数说明：
- `thirdUcId`：三方认证中心编码
- `code`：调用开放平台接口返回的code
- `service`：登录成功后的跳转地址（您的应用地址）

#### 2.3 用户绑定机制

根据文档，用户绑定关系存储在集成认证中心的用户关系表中：
- **首次登录**：用户验证后绑定友户通用户
- **绑定关系**：一个三方用户可以绑定一个友户通用户
- **多个三方用户**：可以绑定同一个友户通用户
- **不支持**：一个三方用户绑定多个友户通帐号

#### 2.4 多因子认证支持

用友BIP支持多种多因子认证方式：

| 认证方式 | Web端/PC端 | 移动端支持 | 支持范围 |
|---------|-----------|-----------|---------|
| 手机短信 | ✅ | ✅ | 大陆、海外 |
| MFA动态码 | ✅ | ✅ | 大陆、海外 |
| 人脸识别 | ✅ | ✅ | 仅支持大陆 |
| CA认证 | ✅ | 禁止登录 | 大陆、海外 |

**建议**：启用手机短信+密码的双重认证，平衡安全性和用户体验。

### 三、用户管理集成方案

#### 3.1 用户模型设计

用友BIP采用**"一个用户-多个身份"模型**：

```
社会化用户（友户通用户）
    ├── 家具企业A
    │   ├── 普通员工身份（绑定员工档案）
    │   ├── 客户管理员身份
    │   └── 商家管理员身份
    ├── 化妆品企业B
    │   └── 普通员工身份
    └── 家具企业C
        └── 普通员工身份
```

**关键概念**：
- **社会化用户**：用友云层面的用户，通过手机号唯一标识
- **租户用户**：企业账号下的用户
- **用户身份**：用户在特定租户下的身份（如普通员工、客户管理员等）
- **员工档案**：HR系统中的员工信息

#### 3.2 用户同步策略

**方案A：实时同步（推荐）**
- 通过用友开放平台API实时获取用户信息
- 优点：数据实时性强
- 缺点：需要调用API，有一定延迟

**方案B：定时同步**
- 定时任务同步用户数据
- 优点：减少API调用
- 缺点：数据有延迟

**方案C：事件驱动同步**
- 监听用友BIP的用户变更事件
- 优点：实时性好，减少不必要的同步
- 缺点：需要配置事件监听

**推荐组合方案**：
- **初始化**：批量同步现有用户
- **日常运维**：事件驱动同步（员工入职/离职）
- **补偿机制**：每日定时全量同步，确保数据一致性

#### 3.3 用户同步关键字段

| 用友BIP字段 | 您的应用字段 | 说明 |
|-----------|------------|------|
| 手机号 | mobile | 国内唯一标识 |
| 邮箱 | email | 国际唯一标识 |
| 用户昵称 | nickname | 社会化属性 |
| 用户名称 | username | 企业内展示名称 |
| 用户ID | yht_user_id | 友户通用户ID |
| 租户ID | tenant_id | 企业账号ID |
| 身份类型 | identity_type | 普通员工/客户管理员等 |
| 员工ID | employee_id | 关联员工档案 |
| 所属组织 | org_id | 用户所属组织 |

#### 3.4 员工入职/离职自动化

用友BIP支持：
- **员工入职自动生成用户**：启用参数"员工启用时是否自动生成/关联用户"
- **员工离职自动移除用户**：员工离职后，关联用户自动移除

**集成建议**：
1. 在您的应用中监听员工入职/离职事件
2. 入职时：调用用友API创建用户，或等待自动生成后同步
3. 离职时：调用用友API移除用户，或在您的应用中标记为离职状态

### 四、权限管理集成方案

#### 4.1 权限模型架构

用友BIP的权限模型包含：

```
权限管理
├── 功能权限（服务、活动按钮）
├── 数据权限（按管理维度授权、按业务授权）
├── 字段权限（读写/只读/无权/禁止）
├── 组织权限（业务单元）
└── 敏感数据权限（金额、客户名称等字段）
```

#### 4.2 角色与授权关系

```
角色
├── 功能权限：可访问的服务和按钮
├── 数据权限：可访问的数据范围
├── 字段权限：可访问的字段
└── 敏感数据权限：敏感字段的访问控制

授权
├── 用户 + 角色 + 组织权限
└── 组织权限 = 数据隔离维度
```

#### 4.3 分级授权模型选择

用友BIP提供两种分级授权模型：

**模型1：组织权限管理员模型（适合中小企业）**
- 按组织分级授权
- 各级单位管理员给本单位的用户授权
- 配置简单，权限宽松
- 启用方式：公共参数 > 启用组织管理员

**模型2：管理员授权模型（适合中大型企业）**
- 按"可管理用户、可管理角色、可授权组织、可授权应用"分级授权
- 更灵活、更严谨
- 启用方式：公共参数 > 启用管理员授权

**推荐方案**：
- 如果您的企业规模较小，组织结构简单：选择**组织权限管理员模型**
- 如果您的企业规模较大，需要精细化权限控制：选择**管理员授权模型**

#### 4.4 权限同步策略

**方案A：角色映射**
- 在您的应用中创建与用友BIP对应的角色
- 用户登录时，根据用友BIP的角色，映射到您应用的角色

**方案B：权限API实时查询**
- 用户访问功能时，实时调用用友BIP权限API
- 优点：权限实时同步
- 缺点：性能开销较大

**方案C：权限缓存+定时刷新**
- 用户登录时，获取并缓存权限信息
- 定时刷新权限缓存
- 优点：性能好，权限相对实时
- 缺点：权限变更有延迟

**推荐方案**：
- **用户登录时**：获取并缓存权限信息
- **权限变更时**：通过事件通知刷新权限
- **补偿机制**：每小时定时刷新权限缓存

#### 4.5 组织权限集成

**重要**：组织权限是数据隔离的关键：
- 业务单据受主组织权限控制
- 只显示组织权限范围内的数据
- 各单据预置了主组织职能类型

**集成建议**：
1. 在您的应用中同步用友BIP的组织架构
2. 业务数据关联组织字段
3. 根据用户的组织权限过滤数据

**3. 权限验证**

```python
# 伪代码示例
from functools import wraps

def check_permission(permission_code):
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            user = session.get('user')
            if not user:
                return redirect('/login')
            
            # 查询用户权限
            permissions = get_user_permissions(user['yht_user_id'])
            
            if permission_code not in permissions:
                return "无权限访问", 403
            
            return f(*args, **kwargs)
        return decorated_function
    return decorator

@app.route('/admin/users')
@check_permission('user:manage')
def manage_users():
    return "用户管理页面"
```

### 七、注意事项与最佳实践

#### 7.1 安全性建议

1. **启用多因子认证**
   - 推荐使用手机短信+密码的双重认证
   - 关键操作启用CA认证

2. **密码策略**
   - 启用租户级密码策略
   - 设置密码复杂度要求
   - 定期强制修改密码

3. **会话管理**
   - 设置合理的会话超时时间
   - 实现单点登出（Single Logout）
   - 记录用户登录日志

4. **权限最小化原则**
   - 按需分配权限
   - 定期审计权限
   - 及时回收离职员工权限

#### 7.2 性能优化建议

1. **缓存策略**
   - 缓存用户权限信息
   - 缓存组织架构数据
   - 设置合理的缓存过期时间

2. **异步处理**
   - 用户同步采用异步方式
   - 权限刷新采用异步方式

3. **批量操作**
   - 批量同步用户
   - 批量同步角色

#### 7.3 容错与降级

1. **API调用失败处理**
   - 实现重试机制
   - 记录失败日志
   - 提供降级方案

2. **数据一致性保障**
   - 定时全量同步
   - 记录同步日志
   - 提供数据修复工具
