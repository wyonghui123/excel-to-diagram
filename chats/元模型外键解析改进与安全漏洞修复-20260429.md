## 第一部分：元模型外键解析改进

### 需求背景

用户要求基于 `BACKLOG-P0-2-MetaModel-Improvement.md` 文档，实施元模型外键解析改进，这是一个核心功能，需要：
1. 细化规格说明
2. 考虑全面影响
3. 补齐自动化测试

### 核心问题

当前系统的元模型设计在父对象关联方面存在不足：
1. **导入时需要用户提供父对象的技术ID而非业务编码** - 用户需要知道 `sub_domain_id=123` 而非 `sub_domain_code="PROCUREMENT"`
2. **验证失败时错误提示不够友好** - 用户无法快速定位问题
3. **跨环境数据迁移需要重新映射ID** - 不同环境ID不一致

### 解决方案

#### Phase 1: Schema 层改进

为所有层级对象添加父对象业务键虚拟字段：

| 对象 | 新增字段 | 父键字段 | resolve_from_field | resolve_to_object |
|------|----------|----------|-------------------|-------------------|
| domain | version_code | version_id | version_code | version |
| sub_domain | domain_code | domain_id | domain_code | domain |
| service_module | sub_domain_code | sub_domain_id | sub_domain_code | sub_domain |
| business_object | service_module_code | service_module_id | service_module_code | service_module |

**修改的文件**:
- `meta/schemas/domain.yaml`
- `meta/schemas/sub_domain.yaml`
- `meta/schemas/service_module.yaml`
- `meta/schemas/business_object.yaml`

#### Phase 2: 服务层改进

在 `action_executor.py` 中添加外键解析方法：

```python
def _resolve_foreign_keys(self, meta_object: MetaObject, data: Dict[str, Any], 
                           original_params: Dict[str, Any] = None) -> Dict[str, Any]:
    """解析外键：从业务键自动解析到技术ID"""
    for field in meta_object.fields:
        resolve_from = getattr(field.semantics, 'resolve_from_field', None)
        resolve_to = getattr(field.semantics, 'resolve_to_object', None)
        
        if resolve_from and resolve_to:
            current_value = data.get(field.id) or data.get(field.db_column)
            source_value = original_params.get(resolve_from) if original_params else data.get(resolve_from)
            
            if (current_value is None or current_value == '') and source_value:
                parent_record = self._find_by_key(resolve_to, 'code', source_value, version_id)
                if parent_record:
                    data[field.db_column] = parent_record.get('id')
                else:
                    raise ValueError(f"父对象 {obj_name} 的业务键 '{source_value}' 不存在...")
    return data
```

## 第二部分：服务器安全漏洞修复

### 漏洞扫描报告分析

用户提供绿盟漏洞扫描报告，显示服务器 `172.20.59.7` 存在大量漏洞：

| 漏洞类别 | 数量 | 风险等级 |
|----------|------|----------|
| Python 相关漏洞 | ~18个 | 中高危 |
| OpenSSH 漏洞 | 2个 | 高危 |
| email 解析器漏洞 | 2个 | 中危 |

### 环境信息澄清

**初始 SOP 记录**: CentOS 7, Python 2.5  
**实际截图显示**: CentOS 7, **Python 3.9.17** (Anaconda 环境)

**关键发现**: Python 3.9.17 已经修复了大部分 CVE 漏洞，但：
1. OpenSSH 仍是 CentOS 7 默认的 7.4 版本（需要升级）
2. 绿盟可能检测到了系统其他位置的旧 Python 实例

### 修复方案

#### 1. OpenSSH 升级（必须）

```bash
# 备份
 cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak.$(date +%Y%m%d)

# 安装编译依赖
yum install -y gcc openssl-devel pam-devel

# 下载并编译 OpenSSH 9.8
cd /opt
wget https://cdn.openbsd.org/pub/OpenBSD/OpenSSH/portable/openssh-9.8p1.tar.gz
tar xzf openssh-9.8p1.tar.gz
cd openssh-9.8p1

./configure \
    --prefix=/usr \
    --sysconfdir=/etc/ssh \
    --with-md5-passwords \
    --with-pam \
    --with-selinux \
    --with-privsep-path=/var/empty/sshd

make && make install

# 重启 SSH 服务
systemctl restart sshd

# 验证
ssh -V  # 应显示 OpenSSH_9.8p1
```

#### 2. Python 升级（推荐）

```bash
# 使用 Anaconda 升级 Python 到 3.11
conda install python=3.11

# 验证
python --version  # 应显示 Python 3.11.x
```

### 修复优先级

| 优先级 | 漏洞 | 操作 | 时间窗口 |
|--------|------|------|----------|
| **P0-紧急** | OpenSSH CVE-2023-48496 | 升级到 9.7+ | 24小时内 |
| **P0-紧急** | OpenSSH CVE-2024-15778 | 升级到 9.8+ | 24小时内 |
| **P1-高** | Python 相关漏洞 | 升级到 3.11+ | 1周内 |
