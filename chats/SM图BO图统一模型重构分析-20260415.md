# SM 图与 BO 图统一模型重构分析

> 聊天日期：2026-04-15
> 主要议题：服务模块图(SM)与业务对象图(BO)的数据模型差异分析、统一重构方向

---

## 一、问题背景

### 1.1 本次修复的问题

SM 图中，当父分组（如"供应链云"Domain）被禁用后，子分组（如"采购供应"SubDomain）的容器（SM 节点）没有正确显示。只有分组标题显示，容器内容为空。

**根本原因**：SM 图中存在"容器层级不匹配"的问题：

| 层级 | GroupModel 中的 containers | realContainers（图表数据） |
|------|---------------------------|--------------------------|
| SM 图 | SubDomain.containers = [SM_Group 对象] | `[{ id: "采购供应", name: "采购供应", nodes: [SM节点...] }]` |

- `SM_Group` 对象的 id 是 `"SM_采购订单管理"` 级别（SM 级别）
- `realContainers` 的 id 是 `"采购供应"`（SubDomain 级别）
- 两者完全不在同一层级，无法匹配

**修复方案**：
1. `convertGroup` 中，SM 图的 SubDomain 的 `group.containers`（SM_Group 对象）不再加入 `legacyGroup.containers`
2. `resolveContainersInGroup` 中，当 group 没有 containers 时，直接用 group 的 name/title 匹配 realContainers，并设置 `directNodes`

### 1.2 第二个问题：多余的嵌套层

修复后容器显示了，但多了一层嵌套（SubDomain subgraph 内部又有一个同名的 container subgraph）。

**修复**：当 group.name 与 matchedContainer.name 完全匹配时，使用 `directNodes` 而不是创建嵌套容器。

## 三、核心问题：containers 的双重含义

### 3.1 两种不同的 containers 含义

| 含义 | 所在层 | 数据结构 | 用途 |
|------|--------|----------|------|
| **含义1** | GroupModel层 | `{ id, type, title, elementRef }` | 表示叶子节点（SM_Group） |
| **含义2** | 渲染层 | `{ id, name, nodes: [{id,name,code}] }` | 表示图表中的子图区域 |

### 3.2 具体表现

**BO 图（全 children）**：
```
Domain.children = [SubDomain]
  SubDomain.children = [SM_Group]
    SM_Group.children = [BO_Group]   ← 终端在 children 里
```

**SM 图（终端用 containers）**：
```
Domain.children = [SubDomain]
  SubDomain.children = []            ← 空！
  SubDomain.containers = [SM_Group]  ← 终端在 containers 里
```

### 3.3 为什么这是问题

GroupModel 中的 containers（SM_Group）和渲染层的 containers（realContainers）是两个不同层级的东西：
- GroupModel.container：SM_Group 对象（SM 级别）
- realContainer：SubDomain 级别的容器（{ id: "采购供应", nodes: [SM1, SM2] }）

这两者无法直接匹配。

## 四、方向A：务实的简化

### 4.1 核心思路

不追求完全统一，而是消除不必要的复杂性：
1. **统一数据结构**：SM 图也用 `children` 承载所有层级
2. **消除 workaround**：移除 `chartType === 'serviceModule' && group.type === 'SUB_DOMAIN'` 判断
3. **清理死代码**：移除 UnifiedRenderer 和 containers 相关死代码

### 4.2 执行计划

**Phase 1：修复 architectureProcessor（低风险）**
- `architectureProcessor.js` 第 214 行：`subDomainGroup.containers.push(smGroup)` → `subDomainGroup.children.push(smGroup)`
- `extractTerminalGroups` 函数：增加自身是否为终端的检查

**Phase 2：清理 GroupModel 死代码（中风险）**
- buildIndex：移除 containers 处理（L124-137）
- getFlattenedGroups：移除 containers 合并（L367-374, L425-434）
- convertGroup：移除 workaround（L605-606）和 containers 块（L601-625）
- toMermaidConfig：移除 containers 提升（L670-688）
- mergeUserGroup：移除 containers 合并（L197-199）

**Phase 3：清理 UnifiedRenderer 死代码（极低风险）**
- MermaidComponent.vue：移除 `&& false` 分支
- useDiagramData.js：移除 UnifiedRenderer 代码生成

### 4.3 为什么改 architectureProcessor 就够了

`convertGroup` 的 children 循环**本来就会跳过终端节点**：
```javascript
if (!childIsTerminal && hasCode) {
  containers.push(convertGroup(child, ...))
}
// 终端子分组被跳过！
```

这意味着：
- BO 图：BO_Group 是终端 → 在 SM_Group 的 children 循环中被跳过
- SM 图（改后）：SM_Group 是终端 → 在 SubDomain 的 children 循环中被跳过

**两种图表走完全相同的逻辑**，终端节点在 convertGroup 中被跳过，由 Syntax 层通过名称匹配来解析。

### 4.4 测试矩阵

| 场景 | SM 图 | BO 图 |
|------|-------|-------|
| 全部启用 | ✅ SubDomain 容器中有 SM 节点 | ✅ SM 容器中有 BO 节点 |
| Domain 禁用 | ✅ SubDomain 被提升，SM 节点通过 directNodes 显示 | ✅ SubDomain 被提升，BO 节点正常显示 |

| 配置面板自动分组 | ✅ SM 正确分配到分组 | ✅ BO 正确分配到分组 |
| 颜色方案 | ✅ 正确着色 | ✅ 正确着色 |
| 注释/备注 | ✅ 正确显示 | ✅ 正确显示 |

### 4.5 量化收益

| 指标 | 改动前 | 改动后 |
|------|--------|--------|
| chart-type-specific 分支 | 10+ 处 | 3-4 处 |

## 五、方向B：基于 BO 图扩展支持 SM 图

### 5.1 核心思路

BO 图已经有完整的 Domain → SubDomain → SM 容器层级，SM 图本质上就是 BO 图的"简化版"——不展示 BO 节点，只展示 SM 容器，并在 SM 容器之间画连线。

### 5.2 两种子方案

#### 子方案 B1：SM 作为 subgraph（真正的 BO 图扩展）

```
┌─ 供应链云 ─────────────────────────────────┐
│ ┌─ 采购供应 ────────────────────────────┐   │
│ │ ┌─ 采购订单管理 ─┐  ┌─ 供应商管理 ─┐  │   │
│ │ │ (BO节点隐藏)   │  │ (BO节点隐藏) │  │   │
│ │ └───────┬────────┘  └──────┬───────┘  │   │
│ └─────────┼──────────────────┼──────────┘   │
└───────────┼──────────────────┼──────────────┘
            │                  │
            │    ┌─────────────┘
            └────┼──────────────────────────────┐
                 │                              │
                 ▼                              ▼
            ┌──────────┐                  ┌──────────┐
            │ SM2 ───→ │                  │ ◄─ SM3   │
            └──────────┘                  └──────────┘
```

- SM 容器保持为 subgraph，BO 节点隐藏
- 连线在 subgraph ID 之间绘制
- **风险**：Mermaid 对嵌套 subgraph 之间连线的渲染质量不稳定
- **风险评级**：**高**

#### 子方案 B2：SM 作为 node（数据复用，渲染转换）

```
┌─ 采购供应（供应链云）─────────────────┐
│                                       │
│   ┌─ 采购订单管理 ─┐  ┌─ 供应商管理 ─┐│
│   │  PO_MGMT       │  │  SUP_MGMT   ││
│   └───────┬────────┘  └──────┬───────┘│
└───────────┼──────────────────┼────────┘
            │     关系          │
            └──────────┬───────┘
                       │
┌─ 库存管理（供应链云）─┼───────────────┐
│                       ▼               │
│   ┌─ 库存管理 ─────────────────────┐  │
│   │  INV_MGMT                      │  │
│   └────────────────────────────────┘  │
└───────────────────────────────────────┘
```

- SM 渲染为**节点**（不是 subgraph），和当前 SM 图一样
- 使用 BO 图的 `moduleGroups` 数据提取 SM 信息
- 使用 `computedServiceModuleRelations()` 生成 SM 级连线
- **视觉效果与当前 SM 图完全相同**
- **风险评级**：**中**

### 5.3 子方案 B2 的可行性验证

## 六、推荐方案

### 6.1 执行顺序建议

1. **Phase 1**：方向A（低风险）
   - 修复 architectureProcessor
   - 清理 GroupModel 死代码
   - 验证两种图表正常工作

2. **Phase 2**：验证方向B可行性
   - 在 useBusinessObjectSyntax.js 中添加 SM 模式原型

3. **Phase 3**：完整实现方向B
   - 移除 SM 专用管线

### 6.2 关键洞察

**方向A的价值**：
- 消除 containers 双重含义
- 简化 GroupModel
- 为方向B铺垫（统一数据结构后，方向B更容易实现）

**方向B的价值**：
- 更大的代码简化（~400行净减少）
- 单一渲染管线维护
- 概念更清晰：一种图表类型，两种模式

**两者不冲突**：方向A是方向B的铺垫，可以分阶段执行。

## 八、已修复的问题记录

### 8.1 SM 图 disabled 父分组后容器不显示

**问题**：Domain 禁用后，SubDomain 的 SM 容器不显示

**原因**：SubDomain.containers 中的 SM_Group 无法匹配 subDomain 级别的 realContainers

**修复**：
1. `convertGroup`：SM 图 SubDomain 的 containers 不加入 legacyGroup
2. `resolveContainersInGroup`：group 无 containers 时用 name 匹配 realContainers，设置 directNodes

### 8.2 多余的嵌套层

**问题**：修复后多了一层同名容器嵌套

**原因**：matchedContainer 创建了新的嵌套 subgraph

**修复**：当 group.name == matchedContainer.name 时，使用 directNodes 避免嵌套

## 十、总结

今天的分析揭示了一个核心问题：**containers 的双重含义**。

- **含义1**：GroupModel 层表示终端子分组
- **含义2**：渲染层表示可视化子图区域

这两个含义被混淆，导致需要 workaround 来处理。方向A通过统一数据结构来消除这个混淆，方向B通过统一渲染管线来彻底解决问题。

**推荐路径**：先执行方向A（低风险），再考虑方向B（更大收益）。
