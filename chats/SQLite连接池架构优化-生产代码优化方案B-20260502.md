## 一、现状分析

| 维度 | 现状 | 问题 |
|------|------|------|
| **连接模型** | 单连接 + `threading.Lock()` 串行化 | 所有操作排队，并发能力为 0 |
| **WAL 模式** | 已启用 | ✅ 良好，但未充分利用多连接并发读 |
| **Checkpoint** | 每 10 次提交自动 truncate | 间隔可能不合理（写密集型场景 WAL 会膨胀） |
| **事务锁** | `_in_transaction` 布尔标志 | 不支持并发事务 |
| **乐观锁** | `update_with_version()` 已实现 | ✅ 良好，对标 SAP ENQUEUE |
| **监控能力** | Prometheus + Grafana 基础设施已有 | ⚠️ 缺少数据库级别指标 |
| **性能基准** | 完善的 PerformanceBenchmark 框架 | ✅ 可用于回归检测 |

## 二、架构设计

### 读写分离连接池架构

```
                    ┌─────────────────────────────┐
                    │     Application Threads      │
                    │   (Flask/Gunicorn workers)   │
                    └──────┬──────────┬────────────┘
                           │          │
                    ┌──────▼──┐  ┌───▼──────────┐
                    │  Write  │  │  Read Pool   │
                    │  Queue  │  │ (N readers)  │
                    │ (1 writer)│  │              │
                    └──────┬──┘  └──┬──────┬─────┘
                           │        │      │
                    ┌──────▼────────▼──────▼─────┐
                    │    SQLite WAL Database     │
                    │   (journal_mode=WAL)       │
                    │   .db  +  .db-wal  + .db-shm│
                    └────────────────────────────┘
```

**设计要点**：
- **写连接独占（1 个）**：写事务必须串行化，通过 `WriteQueue` 确保写入顺序
- **读连接池（N 个，可配置）**：利用 WAL 模式下读不阻塞写的特性，多读者并发
- **连接生命周期管理**：空闲超时回收、最大连接数限制、健康检查
- **向后兼容**：保留 `DataSource` 抽象接口不变，内部实现切换

## 三、实施内容

### Phase 1：连接池基础设施（P0 · 核心）

#### 1.1 SQLiteConnectionPool 连接池类
**文件**：`meta/core/sql_connection_pool.py`

核心功能：
- 连接创建工厂、读写连接分离
- 连接获取/归还（acquire / release）
- 空闲连接回收（idle_timeout）
- 最大连接数控制（max_connections）
- 连接健康检查（ping / is_valid）
- 优雅关闭（close_all / shutdown）

关键参数：
- `max_readers`: 读连接最大数量（默认 5）
- `idle_timeout`: 空闲连接超时秒数（默认 300）
- `max_lifetime`: 连接最大生命周期秒数（默认 3600）
- `acquire_timeout`: 等待获取连接超时秒数（默认 30）

#### 1.2 WriteQueue 串行化写入器
**文件**：`meta/core/sql_write_queue.py`

核心功能：
- 写操作排队（Python Queue / deque + Condition）
- 写事务自动管理（begin / commit / rollback）
- 写操作超时控制
- 写队列深度监控
- 写延迟统计

API：
- `submit(operation_fn, *args, **kwargs)` → Future/Result
- `flush()` → 等待所有挂起写操作完成
- `stats()` → 返回队列统计信息

#### 1.3 重构 SQLiteAdapter 集成连接池
**文件**：`meta/core/sql_adapters.py`（修改）

重构要点：
- 原 `execute()` 方法：`with self._lock: cursor = self._connection.cursor()`
- 重构后：
  - 写操作 → WriteQueue 提交
  - 读操作 → 读连接池获取
- 操作分类逻辑：自动判断 INSERT/UPDATE/DELETE/CREATE 等为写操作

关键特性：
- `use_pool=True`：启用连接池模式（默认）
- `use_pool=False`：降级到 legacy 单连接 + Lock 模式

### Phase 2：WAL 与 Checkpoint 优化（P1 · 稳定性）

#### 2.1 CheckpointManager 智能 Checkpoint 策略
**文件**：`meta/core/sql_checkpoint_manager.py`

策略设计：
- **WAL 大小触发**：`wal_size > wal_size_threshold` → 强制 TRUNCATE checkpoint
- **时间触发**：距上次 checkpoint > `checkpoint_interval_seconds` → TRUNCATE checkpoint
- **低峰期触发**：在请求量低的时间窗口（默认 02:00-05:00）执行 RESTART checkpoint
- **被动模式**：检查是否有活跃读者，有则降级为 PASSIVE checkpoint

配置：
- `wal_size_threshold_mb`: 50   # WAL 文件超过 50MB 触发
- `checkpoint_interval_seconds`: 300  # 5 分钟间隔
- `low_traffic_window_start/end`: 低峰窗口
- `restart_checkpoint_interval_hours`: 6  # 低峰期 RESTART checkpoint 间隔

#### 2.2 WAL 文件大小监控与告警
集成在 CheckpointManager 中：
- `get_wal_size_bytes()`: 获取 WAL 文件当前大小
- `wal_checkpoint_count`: checkpoint 执行次数
- `wal_checkpoint_failures`: checkpoint 失败次数（因活跃读者）

## 四、关键代码实现

### 4.1 读/写操作自动分类

```python
_WRITE_PREFIXES = ('INSERT', 'UPDATE', 'DELETE', 'CREATE', 'DROP',
                   'ALTER', 'REPLACE')

def _classify_operation(sql: str) -> str:
    """分类 SQL 操作类型：read / write"""
    stripped = sql.strip()
    upper = stripped[:30].upper()

    for prefix in _WRITE_PREFIXES:
        if upper.startswith(prefix):
            return 'write'

    if upper.startswith('PRAGMA'):
        if any(kw in upper for kw in ('JOURNAL_MODE', 'WAL_CHECKPOINT',
                                       'SYNCHRONOUS', 'CACHE_SIZE')):
            return 'write'
        return 'read'

    if upper.startswith('SELECT'):
        return 'read'

    return 'read'
```

### 4.2 SQLiteAdapter 自动路由

```python
def execute(self, command: str, params: Optional[tuple] = None) -> Any:
    if self._use_pool and self._pool and self._write_queue:
        op_type = _classify_operation(command)
        if op_type == 'write':
            return self._execute_via_write_queue(command, params)
        else:
            return self._execute_via_read_pool(command, params)
    else:
        return self._execute_legacy(command, params)
```

### 4.3 向后兼容保证

`DataSource` 抽象接口完全不变，现有调用方无需修改任何代码：

```python
class DataSource(ABC):
    # 所有这些方法签名保持不变
    def find_by_id(self, ...)   # 读 → 读连接池
    def find(self, ...)         # 读 → 读连接池

    def update(self, ...)       # 写 → WriteQueue
    def delete(self, ...)       # 写 → WriteQueue
    def execute(self, ...)      # 自动分类路由
    def commit(self)            # 写连接 commit
    def rollback(self)          # 写连接 rollback
```

## 七、风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| WAL 模式下多连接性能未达预期 | 中 | Phase 1 结束后立即运行性能基准测试，对比 baseline |
| 读连接数过多导致文件句柄耗尽 | 低 | `max_readers=5` 保守默认值，监控 `pool_active_readers` |
| WriteQueue 成为新瓶颈 | 低 | 写操作本就是 SQLite 瓶颈，队列化反而减少锁竞争 |
| 兼容性问题影响现有功能 | 中 | 所有现有测试继续通过；保持 DataSource 接口不变 |

## 九、后续优化建议

1. **性能基准测试**：使用 `meta/tests/performance/test_database_performance.py` 对比优化前后的 QPS 和延迟
2. **连接池预热**：启动时预先创建 N 个读连接，避免冷启动延迟
3. **读写分离路由**：根据查询特征（简单查询 vs 复杂分析查询）智能路由到不同连接池
4. **连接池监控告警**：当 `active_readers / max_readers > 80%` 时触发告警，建议扩容
5. **慢查询自动优化建议**：基于 `EXPLAIN QUERY PLAN` 输出自动生成索引建议
