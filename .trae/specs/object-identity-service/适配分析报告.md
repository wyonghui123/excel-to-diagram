## 执行时间
2026-05-09

## 分析范围
全面检查代码库中使用 business_key、parent_key、display_name 的场景，评估是否需要适配新的 ObjectIdentityService。

## 一、需要适配的场景

### 1. 审计日志 API（高优先级）⚠️

**文件**: `meta/api/audit_api.py`

**现状**:
- 使用硬编码的 `BUSINESS_KEY_METADATA` 字典配置
- 使用 `_generate_business_key()` 函数生成业务键
- 仅用于审计日志展示

**问题**:
- ❌ 硬编码配置，非元数据驱动
- ❌ 功能单一，不支持层级路径
- ❌ 无法复用到其他场景

**适配方案**:
```python
# 旧代码（需要废弃）
def _generate_business_key(data_source, object_type, object_id, ...):
    meta = BUSINESS_KEY_METADATA.get(object_type)
    # ... 硬编码逻辑

# 新代码（推荐）
from meta.services.object_identity_service import ObjectIdentityService

def _generate_business_key(data_source, object_type, object_id, ...):
    """
    @deprecated 使用 ObjectIdentityService 替代
    保留此函数以保持向后兼容
    """
    service = ObjectIdentityService(data_source)
    identity = service.get_identity(object_type, object_id, format='short')
    return identity.get('formatted', '')
```

**影响范围**: 审计日志列表展示、审计日志详情

---

### 2. 导入导出服务（中优先级）⚠️

**文件**: `meta/services/import_export_service.py`

**现状**:
- 使用 `parent_key_columns` 标识父对象业务键列
- 使用 `business_key` 字段进行唯一性校验
- 导入时解析父级上下文

**问题**:
- ⚠️ 部分功能与 ObjectIdentityService 重叠
- ⚠️ 但导入导出有特殊需求（如 Excel 列标识）

**适配方案**:
- **不建议立即迁移**
- 导入导出服务有自己的业务逻辑（Excel 列、导入顺序等）
- 可以在后续版本中考虑复用 HierarchyPathService

---

### 3. 前端审计日志页面（高优先级）⚠️

**文件**: `src/views/SystemManagement/AuditLogManagement.vue`

**现状**:
- 从后端获取 business_key 字段
- 直接展示，无前端格式化

**适配方案**:
```javascript
// 引入 Composable
import { useObjectIdentity } from '@/composables/useObjectIdentity'

const { getFormattedIdentity } = useObjectIdentity()

// 在数据加载后获取对象标识
async function loadObjectIdentities(logs) {
  const requests = logs.map(log => ({
    object_type: log.object_type,
    object_id: log.object_id
  }))
  
  const identities = await batchGetIdentities(requests, { format: 'short' })
  
  // 更新 logs，添加 formatted_identity 字段
  logs.forEach(log => {
    const key = `${log.object_type}:${log.object_id}:short:false`
    log.formatted_identity = identities.get(key)?.formatted || log.business_key
  })
}
```
