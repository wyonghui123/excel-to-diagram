## 一、适配场景分类

### 1. 审计日志场景（P0 - 高优先级）⚠️

#### 1.1 审计日志 API
**文件**: `meta/api/audit_api.py`

**现状**:
- 使用硬编码的 `BUSINESS_KEY_METADATA` 字典配置
- 使用 `_generate_business_key()` 函数生成业务键
- 仅用于审计日志展示

**问题**:
- ❌ 硬编码配置，非元数据驱动
- ❌ 功能单一，不支持层级路径
- ❌ 无法复用到其他场景

**适配方案**:
```python
# 迁移方案
from meta.services.object_identity_service import ObjectIdentityService

def _generate_business_key(data_source, object_type, object_id, ...):
    """
    @deprecated 使用 ObjectIdentityService 替代
    保留此函数以保持向后兼容
    """
    service = ObjectIdentityService(data_source)
    identity = service.get_identity(object_type, object_id, format='short')
    return identity.get('formatted', '')
```

**影响范围**: 
- 审计日志列表展示
- 审计日志详情
- 审计日志导出

---

#### 1.2 前端审计日志页面
**文件**: `src/views/SystemManagement/AuditLogManagement.vue`

**现状**:
- 从后端获取 business_key 字段
- 直接展示，无前端格式化

**适配方案**:
```javascript
// 引入 Composable
import { useObjectIdentity } from '@/composables/useObjectIdentity'

const { batchGetIdentities } = useObjectIdentity()

// 在数据加载后获取对象标识
async function loadObjectIdentities(logs) {
  const requests = logs.map(log => ({
    object_type: log.object_type,
    object_id: log.object_id
  }))
  
  const identities = await batchGetIdentities(requests, { format: 'short' })
  
  // 更新 logs，添加 formatted_identity 字段
  logs.forEach(log => {
    const key = `${log.object_type}:${log.object_id}:short:false`
    log.formatted_identity = identities.get(key)?.formatted || log.business_key
  })
}
```

### 3. 架构数据管理场景（P2 - 低优先级）✅

#### 3.1 详情面板
**文件**: `src/views/ArchDataManageApp/components/DetailPanel.vue`

**现状**:
- 展示 domain_name、sub_domain_name、service_module_name 等字段
- 通过 hierarchyPath 计算属性构建层级路径
- 这些字段由 enrichment_engine 填充

**架构分析**:
```
后端流程：
1. query_service.py 查询数据
2. enrichment_engine.py 填充虚拟冗余字段（domain_name, sub_domain_name 等）
3. redundancy_registry.py 定义虚拟冗余字段规则

前端流程：
1. DetailPanel.vue 接收数据
2. hierarchyPath 计算属性构建层级路径
3. 展示完整的层级路径
```

**问题**:
- ✅ 现有架构已经很好地解决了层级路径展示问题
- ✅ enrichment_engine 提供了元数据驱动的虚拟字段填充
- ✅ 前端通过计算属性灵活构建路径

**适配方案**:
- **不建议立即迁移**
- 现有架构已经很好地解决了问题
- enrichment_engine 与 ObjectIdentityService 功能重叠
- 可以考虑在后续版本中整合

**对比分析**:

| 维度 | EnrichmentEngine | ObjectIdentityService |
|------|------------------|----------------------|
| 设计目标 | 查询时填充虚拟冗余字段 | 统一对象标识转换 |
| 实现方式 | 从 redundancy_registry 读取定义 | 从元数据读取 business_key 定义 |
| 适用场景 | 列表查询、详情展示 | 审计日志、通用标识转换 |
| 性能 | 批量查询优化 | 批量查询优化 + 缓存 |
| 层级路径 | 需要前端计算 | 后端直接返回 |

**建议**:
1. **短期**: 保持现有架构，ObjectIdentityService 用于审计日志等新场景
2. **中期**: 评估两者整合的可能性
3. **长期**: 考虑统一对象标识服务

## 三、技术架构对比

### 3.1 EnrichmentEngine vs ObjectIdentityService

#### EnrichmentEngine 架构
```
元数据定义（YAML）
  ↓
redundancy_registry.py（注册表）
  ↓
enrichment_engine.py（填充引擎）
  ↓
query_service.py（查询服务）
  ↓
前端组件（DetailPanel.vue）
```

#### ObjectIdentityService 架构
```
元数据定义（YAML）
  ↓
business_key_service.py（业务键服务）
hierarchy_path_service.py（层级路径服务）
  ↓
object_identity_service.py（统一标识服务）
  ↓
object_identity_api.py（RESTful API）
  ↓
前端 Composable（useObjectIdentity.js）
```

#### 功能对比

| 功能 | EnrichmentEngine | ObjectIdentityService |
|------|------------------|----------------------|
| 虚拟字段填充 | ✅ | ❌ |
| 业务键转换 | ❌ | ✅ |
| 层级路径计算 | ❌（前端计算） | ✅（后端计算） |
| 元数据驱动 | ✅ | ✅ |
| 批量查询优化 | ✅ | ✅ |
| 缓存机制 | ✅（内部缓存） | ✅（多级缓存） |
| API 接口 | ❌ | ✅ |
| 前端 Composable | ❌ | ✅ |
