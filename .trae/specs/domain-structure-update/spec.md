# 领域结构更新 - 产品需求文档

## Overview
- **Summary**: 更新系统中的领域结构命名，将领域产品变更为领域，产品模块变更为子领域，产品子模块变更为服务模块，并相应调整Excel/CSV导入的列名称和预览页面的展示列名称。
- **Purpose**: 统一和优化领域结构的命名规范，使其更加清晰和符合业务术语。
- **Target Users**: 系统用户、数据分析师、业务用户

## Goals
- 将领域产品变更为领域
- 将产品模块变更为子领域
- 将产品子模块变更为服务模块
- 调整Excel/CSV导入的列名称以匹配新的命名
- 更新预览页面的展示列名称
- 保持系统功能的完整性和兼容性

## Non-Goals (Out of Scope)
- 改变系统的核心功能和逻辑
- 修改图表渲染引擎的行为
- 变更业务对象和业务对象关系的基本结构

## Background & Context
当前系统使用的命名规范可能导致混淆，需要统一为更清晰的术语。领域结构的层次关系应为：领域 → 子领域 → 服务模块 → 业务对象。

## Functional Requirements
- **FR-1**: 将领域产品变更为领域
- **FR-2**: 将产品模块变更为子领域
- **FR-3**: 将产品子模块变更为服务模块
- **FR-4**: 调整Excel/CSV导入的列名称以匹配新的命名
- **FR-5**: 更新预览页面的展示列名称
- **FR-6**: 确保系统能正确识别新的列名称

## Non-Functional Requirements
- **NFR-1**: 保持系统的向后兼容性，确保旧的列名称仍能被识别
- **NFR-2**: 确保系统性能不受影响
- **NFR-3**: 确保用户界面的一致性和清晰度

## Constraints
- **Technical**: 系统使用Vue 3框架，需要修改前端模板和数据处理逻辑
- **Business**: 变更应保持与现有业务流程的兼容性

## Assumptions
- 新的命名规范将被所有用户接受
- 现有的数据结构和关系逻辑保持不变
- 系统能够正确识别和处理新的列名称

## Acceptance Criteria

### AC-1: 领域结构命名更新
- **Given**: 系统已更新
- **When**: 用户查看系统界面
- **Then**: 所有显示"领域产品"的地方都显示"领域"，"产品模块"显示"子领域"，"产品子模块"显示"服务模块"
- **Verification**: `human-judgment`

### AC-2: Excel/CSV导入列名称调整
- **Given**: 用户上传包含新列名称的Excel/CSV文件
- **When**: 系统解析文件
- **Then**: 系统能正确识别新的列名称（领域、子领域、服务模块）
- **Verification**: `programmatic`

### AC-3: 预览页面展示调整
- **Given**: 用户上传文件后查看预览
- **When**: 预览页面加载
- **Then**: 预览页面的列名称显示为新的命名（领域、子领域、服务模块）
- **Verification**: `human-judgment`

### AC-4: 向后兼容性
- **Given**: 用户上传包含旧列名称的Excel/CSV文件
- **When**: 系统解析文件
- **Then**: 系统仍能正确识别旧的列名称（领域产品、产品模块、产品子模块）
- **Verification**: `programmatic`
