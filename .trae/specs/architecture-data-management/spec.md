# Spec: 架构数据管理模块

## 1. Background & Objectives

### 1.1 背景

现有元数据模型已定义了完整的架构元素层级结构：产品(product) → 版本(version) → 领域(domain) → 子领域(sub_domain) → 服务模块(service_module) → 业务对象(business_object)，以及业务关系(relationship)。然而，当前缺乏统一的管理界面和查询系统，导致：

1. 数据维护效率低，需要手动操作数据库
2. 数据一致性难以保障，缺乏约束检查
3. 操作无法追溯，缺乏审计机制

### 1.2 业务目标

- 提供统一的架构数据管理界面，提升数据维护效率
- 建立完善的数据一致性保障机制
- 实现操作审计追溯能力

### 1.3 用户/涉众目标

- **架构师**: 高效管理架构元素，确保数据一致性
- **开发人员**: 快速查询和浏览架构数据

## 2. Requirement Type Overview

| 类型 | 适用 | 证据来源 |
|-----|-----|---------|
| 业务需求 | 是 | 提升架构数据管理效率 |
| 用户/涉众需求 | 是 | 架构师、开发人员使用场景 |
| 解决方案需求 | 是 | 元数据模型驱动的设计方案 |
| 功能需求 | 是 | 查询、管理、审计功能 |
| 非功能需求 | 是 | 性能、可用性要求 |
| 外部接口需求 | 是 | RESTful API、前端界面 |
| 过渡需求 | 是 | 数据迁移 |

## 3. Functional Requirements

### FR-001: 多条件组合查询

- **描述**: 系统必须支持对架构元素进行多条件组合查询，包括精确匹配、模糊搜索、范围查询等
- **验收标准**:
  - 支持AND/OR条件组合
  - 支持LIKE模糊搜索（不区分大小写）
  - 支持IN、BETWEEN范围查询
  - 支持分页和排序
  - 查询响应时间 < 500ms（1000条记录内）
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 用户需求分析

### FR-002: 层级路径查询

- **描述**: 系统必须支持按层级路径查询架构元素，如"产品/版本/领域/子领域"
- **验收标准**:
  - 支持路径格式: "供应链系统/v1.0/供应链领域/采购管理"
  - 支持通配符查询: "供应链系统/*"
  - 支持查询指定节点的所有子节点
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 架构师使用场景

### FR-003: 全文检索

- **描述**: 系统必须支持跨对象类型的全文检索功能
- **验收标准**:
  - 支持跨业务对象、服务模块、领域等搜索
  - 支持按对象类型过滤
  - 返回结果按相关性排序
  - 支持搜索建议（自动补全）
- **优先级**: Should
- **类型映射**: 功能需求
- **来源**: 用户需求分析

### FR-004: CRUD操作管理

- **描述**: 系统必须提供完整的CRUD操作能力，包括创建、读取、更新、删除架构元素
- **验收标准**:
  - 支持单条记录的CRUD操作
  - 创建/更新时自动执行校验规则
  - 删除前检查引用完整性
  - 支持软删除和硬删除
  - 操作结果返回明确的状态信息
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 架构师使用场景

### FR-005: 批量操作

- **描述**: 系统必须支持批量创建、更新、删除操作
- **验收标准**:
  - 支持批量导入Excel数据
  - 支持批量更新多个记录
  - 支持批量删除（带确认）
  - 返回详细的操作结果（成功数、失败数、错误详情）
- **优先级**: Should
- **类型映射**: 功能需求
- **来源**: 架构师使用场景

### FR-006: 数据导入导出

- **描述**: 系统必须支持架构数据的Excel导入导出
- **验收标准**:
  - 支持按模板格式导入
  - 导入时自动校验数据
  - 支持导出查询结果
  - 导出文件包含完整字段信息
- **优先级**: Should
- **类型映射**: 功能需求
- **来源**: 用户需求分析

### FR-007: 层级关系约束

- **描述**: 系统必须在创建/更新时检查层级关系约束
- **验收标准**:
  - 子对象必须存在有效的父对象
  - 父对象必须属于同一个版本
  - 层级路径必须连续
  - 违反约束时返回明确错误信息
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 数据一致性需求

### FR-008: 引用完整性检查

- **描述**: 系统必须在删除时检查引用完整性
- **验收标准**:
  - 删除前检查被引用情况
  - 有引用时提供选项：禁止删除/级联删除/置空引用
  - 业务关系随业务对象级联删除
  - 循环引用检测
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 数据一致性需求

### FR-009: 唯一性约束

- **描述**: 系统必须保证业务键和编码的唯一性
- **验收标准**:
  - 业务对象编码在版本范围内唯一
  - 领域编码在版本范围内唯一
  - 重复时返回明确错误信息
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 数据一致性需求

### FR-010: 操作审计日志

- **描述**: 系统必须记录所有CRUD操作的审计日志
- **验收标准**:
  - 记录操作类型、时间、用户、IP地址
  - 记录变更前后的数据对比
  - 支持按对象、用户、时间查询日志
  - 审计日志不可修改和删除
- **优先级**: Must
- **类型映射**: 功能需求
- **来源**: 合规需求

### FR-011: 变更历史追踪

- **描述**: 系统必须支持查看单个对象的变更历史
- **验收标准**:
  - 显示所有变更记录

  - 显示变更字段和值对比
  - 支持导出变更历史
- **优先级**: Should
- **类型映射**: 功能需求

## 4. Nonfunctional Requirements

### NFR-001: 性能要求

- **描述**: 系统必须满足以下性能指标
- **测量方法**:
  - 单条记录查询响应时间 < 100ms
  - 列表查询（1000条内）响应时间 < 500ms
  - 批量操作（100条）响应时间 < 5s
- **优先级**: Must
- **来源**: 性能需求

### NFR-002: 可用性要求

- **描述**: 系统必须满足以下可用性要求
- **测量方法**:
  - 错误提示清晰明确
  - 操作可撤销（软删除）
  - 界面响应流畅
- **优先级**: Should
- **来源**: 可用性需求

### NFR-003: 可维护性要求

- **描述**: 系统必须满足以下可维护性要求
- **测量方法**:
  - 代码覆盖率 >= 80%
  - API文档完整
  - 日志记录完整
- **优先级**: Should
- **来源**: 可维护性需求

## 5. External Interface Requirements

### IF-001: 查询API

- **类型**: API
- **端点**: 
  - `POST /api/v1/query/search` - 通用搜索
  - `GET /api/v1/query/full-text` - 全文检索
  - `GET /api/v1/query/hierarchy/{path}` - 层级路径查询
  - `GET /api/v1/query/suggest/{type}/{field}` - 自动补全
- **请求/响应**:
  ```json
  // POST /api/v1/query/search
  {
    "object_type": "business_object",
    "conditions": [
      {"field": "code", "operator": "like", "value": "BO_%"}
    ],
    "keyword": "订单",
    "page": 1,
    "page_size": 20
  }
  // Response
  {
    "success": true,
    "data": [...],
    "total": 100,
    "page": 1,
    "page_size": 20
  }
  ```
- **错误处理**: 400参数错误, 500服务器错误
- **来源**: API设计

### IF-002: 管理API

- **类型**: API
- **端点**:
  - `POST /api/v1/{type}` - 创建
  - `GET /api/v1/{type}/{id}` - 查询单条
  - `GET /api/v1/{type}` - 列表查询
  - `PUT /api/v1/{type}/{id}` - 更新
  - `DELETE /api/v1/{type}/{id}` - 删除
  - `POST /api/v1/{type}/batch-create` - 批量创建
  - `POST /api/v1/{type}/import` - 导入
  - `GET /api/v1/{type}/export` - 导出
- **来源**: API设计

### IF-003: 审计API

- **类型**: API
- **端点**:
  - `GET /api/v1/audit/logs` - 查询审计日志
  - `GET /api/v1/audit/object/{type}/{id}` - 对象变更历史
- **来源**: API设计

## 7. Constraints & Assumptions

### 7.1 技术约束

- 基于现有元数据模型和QueryBuilder/ActionExecutor基础设施
- 使用Python Flask作为后端框架
- 使用SQLite作为数据库（可扩展其他数据库）
- 前端使用Vue.js或React

### 7.2 业务约束

- 层级结构固定：产品→版本→领域→子领域→服务模块→业务对象
- 业务对象编码格式：BO_前缀
- 软删除为默认删除方式

### 7.3 假设

- 审计日志表已存在
- 用户已通过身份认证系统登录

## 9. Change / Design Proposal (RFC)

### 9.1 As-Is 分析

- **当前架构**: 
  - 元数据模型定义完整（MetaObject, MetaField, MetaRelation等）
  - QueryBuilder支持链式查询构建
  - ActionExecutor支持CRUD操作和规则执行
  - AuditLogger支持基础审计日志记录
  - RuleEngine支持校验、计算、触发规则

- **当前问题**:
  - 缺乏统一的查询服务封装
  - 缺乏批量操作支持
  - 审计功能不够完善
  - 缺乏前端界面

- **相关代码路径**:
  - `meta/core/query_builder.py` - 查询构建器
  - `meta/core/action_executor.py` - 操作执行器
  - `meta/core/rule_executor.py` - 规则引擎
  - `meta/schemas/*.yaml` - 元模型定义

#### 9.3.2 模块/组件设计

**QueryService** - 查询服务
- 职责：封装QueryBuilder，提供统一查询接口
- 主要方法：
  - `search(request: SearchRequest) -> SearchResult`
  - `full_text_search(keyword, types, limit) -> Dict`
  - `query_by_hierarchy_path(path, include_children) -> List`
  - `suggest(object_type, field, prefix, limit) -> List`

**ManageService** - 管理服务
- 职责：扩展ActionExecutor，支持批量操作和导入导出
- 主要方法：
  - `create(request: CreateRequest) -> ActionResult`
  - `update(request: UpdateRequest) -> ActionResult`
  - `delete(request: DeleteRequest) -> ActionResult`
  - `batch_create(object_type, data_list) -> BatchOperationResult`
  - `import_from_excel(object_type, file_path) -> BatchOperationResult`
  - `export_to_excel(object_type, filters) -> str`

**ConsistencyService** - 一致性服务
- 职责：检查数据一致性约束
- 主要方法：
  - `check_hierarchy_constraint(object_type, data) -> ConsistencyCheckResult`
  - `check_reference_integrity(object_type, data) -> ConsistencyCheckResult`
  - `check_uniqueness(object_type, data) -> ConsistencyCheckResult`
  - `validate_full(object_type, data) -> ConsistencyCheckResult`

**AuditService** - 审计服务
- 职责：审计日志查询和分析
- 主要方法：
  - `query(query: AuditQuery, page) -> Dict`
  - `get_object_history(object_type, object_id) -> List`
  - `get_user_activities(user_id, days) -> Dict`
  - `get_change_summary(object_type) -> Dict`

**CascadeService** - 级联服务
- 职责：处理级联删除和更新
- 主要方法：
  - `get_cascade_strategy(parent_type, child_type) -> CascadeStrategy`
  - `before_delete(object_type, id) -> Dict`
  - `execute_cascade(actions) -> Dict`

### 9.4 备选方案

| 方案 | 优点 | 缺点 | 决策 |
|-----|------|------|-----|
| 方案A: 独立服务 | 解耦清晰，可独立部署 | 增加运维复杂度 | 拒绝 |
| 方案B: 集成到现有模块 | 复用现有基础设施，开发效率高 | 模块职责可能膨胀 | 选择 |
| 方案C: 微服务架构 | 高度可扩展 | 过度设计，增加复杂度 | 拒绝 |

选择方案B的理由：
1. 现有QueryBuilder和ActionExecutor已提供良好基础
2. 开发周期更短
3. 运维复杂度低
4. 满足当前需求规模

### 9.5 实施与迁移计划

#### 实施顺序

1. **第一阶段（核心功能）**
   - 实现QueryService
   - 实现ManageService（扩展ActionExecutor）
   - 实现ConsistencyService
   - 实现CascadeService

2. **第二阶段（增强功能和用户界面）**
   - 实现AuditService（扩展AuditLogger）
   - 实现导入导出功能
   - 实现前端用户界面

3. **第三阶段（高级功能）**
   - 实现关系可视化
   - 性能优化

#### 风险缓解

| 风险 | 缓解策略 |
|-----|---------|
| 性能瓶颈 | 添加查询缓存，优化SQL |
| 数据迁移失败 | 提供回滚脚本，增量迁移 |
| 并发冲突 | 乐观锁，版本控制 |

#### 测试策略

- **单元测试**: 覆盖所有Service方法，覆盖率 >= 80%
- **集成测试**: 测试完整业务流程
- **E2E测试**: 测试关键用户场景

#### 回滚计划

1. 保留原有API不变
2. 新增API通过版本号区分（/api/v1/）
3. 数据库变更使用增量脚本
4. 提供数据迁移回滚脚本
