## 强制检查清单

**【警告】如果未完成以下检查，规范将无法生效！**

### 样式修改前必须回答的问题

- [ ] **1. 是否查阅了 `src/styles/YON_DESIGN_CONSTANTS.md`？**
  - 主色调是否为 YonDesign Orange (#ea580c)？
  - 按钮状态是否符合规范（Link 按钮 / Filled 按钮）？
  - 是否使用了 CSS 变量而非硬编码颜色？

- [ ] **2. 是否查阅了 `src/styles/YON_EP_GUIDE.md`？**
  - 是否使用了 `AppModal` 而非 `el-dialog`？
  - 是否使用了 `AppButton` 而非 `el-button`？

- [ ] **3. 代码中是否有 Emoji？**
  - 代码、注释、文档中是否使用了 Emoji？
  - 如有，必须立即替换为 `[WARNING]`、`[OK]`、`[X]` 等文本标记

- [ ] **4. 是否在组件对比页面验证了效果？**
  - http://localhost:3004/component-comparison
  - 按钮的 Hover 状态是否符合规范？

### 违规后果

- **[X] 代码需要重新修改**，浪费开发时间
- **[X] 用户需要重新测试**，浪费测试时间
- **[X] AI 的可信度下降**，用户信任度降低

### 如何确保检查被执行

1. **在开始修改前**，先读取 `src/styles/YON_DESIGN_CONSTANTS.md`
2. **修改完成后**，对比组件对比页面的效果
3. **提交前**，使用 `grep` 命令检查是否有 Emoji 和硬编码颜色

## 会话间一致性保证机制

### 问题描述

在多会话开发中，经常出现以下问题：

- 新会话开始时，AI 不记得之前的规范
- 规范文件存在，但 AI 没有读取
- 规范被忽略，导致代码需要重新修改

### 解决方案

#### 1. 会话开始确认机制（强制执行）

**在每个新会话的第一条消息中，AI 必须：**

```
【会话开始确认】

我已阅读以下规范文件：
- [ ] `.trae/rules/SESSION_REMINDER.md` （会话开始提醒）
- [ ] `src/styles/YON_DESIGN_CONSTANTS.md` （设计规范）
- [ ] `src/styles/YON_EP_GUIDE.md` （组件规范）

我将遵守以下规范：
- [ ] 主色调：YonDesign Orange (#ea580c)
- [ ] 按钮：使用 AppButton，禁止 el-button
- [ ] 弹窗：使用 AppModal，禁止 el-dialog
- [ ] 颜色：使用 CSS 变量，禁止硬编码
- [ ] Emoji：禁止使用任何 Emoji 符号

如果任务涉及 UI/样式，我将：
- [ ] 在开始前读取设计规范
- [ ] 在修改后验证组件对比页面
- [ ] 使用 grep 检查是否有遗漏
```

#### 2. 规范引用机制（强制执行）

**在每次修改样式前，必须：**

1. **引用规范文件**：在回复中包含类似以下的声明
   ```
   [规范检查] 我已查阅 src/styles/YON_DESIGN_CONSTANTS.md，
   确认使用 #ea580c 作为主色调，使用 CSS 变量而非硬编码。
   ```

2. **标记检查清单**：在回复中标记 `[OK]` 表示已完成检查
   ```
   - [OK] 主色调检查
   - [OK] 按钮组件检查
   - [OK] Emoji 检查
   ```

3. **验证效果**：在回复中包含验证说明
   ```
   [验证] 已在 http://localhost:3004/component-comparison 验证效果
   ```

#### 3. 会话间上下文传递机制（建议执行）

**如果当前会话需要承接上一个会话的工作：**

1. **读取项目状态文件**：
   ```bash
   cat .trae/memory/project-status.md
   ```

2. **读取设计决策清单**：
   ```bash
   cat src/styles/DESIGN_CHECKLIST.md
   ```

3. **在回复中声明继承**：
   ```
   [上下文继承] 我已读取项目状态和设计决策清单，
   将继续遵循之前的规范，并在必要时更新设计决策清单。
   ```

### 违规检测和纠正流程

#### 检测阶段

**如果发现可能违反规范的情况，必须：**

1. **立即停止**：暂停当前修改
2. **检查规范**：读取相关的规范文件
3. **自我纠正**：修正可能违反规范的代码
4. **向用户说明**：解释违反的原因和纠正措施

#### 纠正示例

**错误做法**：
```
[不推荐] 直接修改代码，忽略规范检查
```

**正确做法**：
```
[推荐] 我注意到这可能违反设计规范，让我先检查一下...

[规范检查] 我已查阅 src/styles/YON_DESIGN_CONSTANTS.md，
发现我的修改可能违反了颜色使用规范。

[自我纠正] 我已将硬编码的 #1677ff 替换为
var(--yonyou-orange-600)，并添加了必要的注释。

[向用户说明] 我发现并纠正了一个潜在问题，
现在代码符合规范要求。
```
