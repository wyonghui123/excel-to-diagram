## 三、合规/审计视角（运维/合规）

### 3.1 [CRITICAL] 仍存在的合规风险

#### P1-1: D.2 修复未完全覆盖（12 条残留）

| 字段 | 数值 |
|------|------|
| 残留数量 | 12 条 |
| 格式 | `'Admin (admin)'` |
| 产生时间 | 2026-06-19 18:37-18:42（D.2 修复**之后**）|
| User-Agent | `python-requests/2.33.1` |
| 来源 | 某个测试脚本（不是浏览器）|
| 3 个 transaction_id | 各 4 条记录 |

**业务影响**：业务人员看到 `'Admin (admin)'` 会困惑到底是谁
**根因**：D.2 修复了在 `meta/core/audit_constants.py` 的 `normalize_user_name`，但某些测试脚本可能直接调用 `context.user_name = "Admin (admin)"`，**不经过 normalize_user_name**。
**建议**：
1. 调查测试脚本的入口路径（找到这个 python-requests 调用点）
2. 或者：在 AuditInterceptor 里强制 normalize，不允许"display (username)"格式入库

#### P1-2: row_hash 链式哈希未激活（合规风险）

| 字段 | 数值 |
|------|------|
| row_hash 覆盖 | **0/3222 (0%)** |
| prev_hash 覆盖 | **0/3222 (0%)** |
| 全表统计 | 36/20443 (0.18%) |

**业务影响**：
- 审计日志可以被人修改而不被发现
- 不满足金融/医疗等行业的"不可篡改"合规要求
- 数据库层面没有完整性保护

**建议**：
- 数据库表结构已经有 `row_hash` + `prev_hash` 字段
- 但写入路径没有调用哈希计算
- 建议在 AuditInterceptor.after_action() 里增加 hash 链计算

#### P2-1: transaction_id 覆盖率仅 7.1%（可追溯性差）

| 字段 | 数值 |
|------|------|
| 有 tx_id | 230/3222 (7.1%) |
| 无 tx_id | 2992/3222 (92.9%) |
| 其中 system 用户 | 324 (100% 无) |
| 其中 Admin 无 tx | 2492/2662 (93.6%)|

**业务影响**：
- 业务人员问"**这次保存改了哪些字段？**" → 无法回答
- 业务人员问"**这笔业务操作的完整链路？**" → 无法回答
- 3222 条记录中只有 48 个独立事务，平均 67 条/事务（**但实际上只有 7% 有事务**）

**根因**：
- C.1 修复在 `meta/core/action_executor.py` 里自动生成 tx_id
- 但**种子脚本、迁移脚本、Excel 导入脚本**走的是另一条路径，不经过 ActionExecutor
- 这些路径不会生成 tx_id

**建议**：
- 在所有数据写入入口（包括 seed scripts、import scripts）也生成并传递 tx_id
- 或者：在 audit_logs 表加一个数据库 trigger，自动设置 tx_id 如果为空
