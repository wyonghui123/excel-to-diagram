3 · 入门：5 步加一个新枚举值

  1. 进入「系统配置 → 枚举管理」。
  2. 搜索目标枚举（如 order_status）。
  3. 进入详情，点「+ 新增值」。
  4. 填写 code（如 PENDING_REVIEW）、label、上线日期。
  5. 提交后等待代码评审 + CI 通过，自动同步到 dev/staging/prod 三环境。
原则：code 不可修改、label 可修改；下线用「停用」而非删除。

4 · 探索：三件套对照

配置类型存储位置典型用途修改权限
枚举YAML → DB cache状态码、类型字典配置管理员
菜单YAML + 角色绑定左侧导航 + 权限点系统管理员
用户组DB + 角色绑定把人按职责聚合组织管理员

5 · 进阶：灰度与回滚

  • 灰度发布：新枚举值可设置"仅 dev 可见"或"灰度 10% 用户可见"。
  • 配置回滚：每次发布生成快照，一键回到任意历史版本。
  • YAML diff：UI 改动自动生成 PR，可在 GitLab / GitHub 上 review。
小心：已使用的枚举值被停用，会让历史数据展示为「未知」——必须先迁数据再停用。

6 · 实战：线上事故复盘

事故：某次运营活动新增 PROMO_ACTIVE 状态，没走审批直接上线，导致订单流程分支异常。

步骤操作结果
1审计日志锁定操作人 + 时间trace_id = e2c8...a14
2YAML diff 查看新增值确认是新增非修改
3一键回滚到上一个 release5 秒恢复
4补走审批 + 加 lint 规则同类问题不再发生
5事后复盘报告归档合规留痕
